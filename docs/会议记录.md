# 会议记录

## 会议 1：迭代三方向确认

时间：2026-05-12
参与者：全员
主题：确定产品主线和交付范围

### 讨论内容

- 迭代三需要同时交付代码、产品化文档、Agent 优化记录和优化数据集。
- 原有“只让 Agent 输出行程文本”的形态不足以体现产品化价值，用户还需要继续编辑和复盘。
- 旅行规划适合沉淀用户风格，因此 Skill 不应只作为提示词，而应成为可导入、可发布、可复用的资产。
- 普通用户主界面不应暴露主/子 Agent、trace、tool call 等内部实现词。

### 会议结论

- 产品主线确定为“旅行规划工作台 + 旅行风格 Skill 广场 + 跨对话 Agent 记忆”。
- Agent 输出必须写入结构化行程画布，并在助手回复末尾展示本轮改动。
- Skill 需要支持导入、抽取、创建、发布、收藏和添加到当前行程。
- 交付文档需要特别说明用户如何管理对话、知识如何跨对话共享、用户如何向 Agent 添加 Skill。

## 会议 2：架构和数据模型确认

时间：2026-05-17
参与者：全员
主题：确定前后端分层、共享领域层和持久化方案

### 讨论内容

- 行程数据包含日期、活动、地点、路线、偏好、Skill、天气和导出检查，前后端需要共用类型。
- Agent 产生的是结构化 patch，不应让前端从自然语言中猜测修改。
- SQLite 足够支撑本地开发和验收，但需要明确保存行程、Skill、会话、trace 和偏好记忆。
- 评估数据集是交付产物，不一定需要做成运行时 API。

### 会议结论

- 使用 TypeScript monorepo：`apps/web`、`apps/api`、`packages/shared`。
- 共享层负责行程编辑、路线风险、Skill 解析、导出检查等领域逻辑。
- 后端使用 SQLite 保存 `itineraries`、`skills`、`sessions`、`traces`、`skill_creator_sessions`、`memories`。
- 评估数据集以 JSON 文件归档在 `data/evaluation/agent-optimization-dataset.json`。

## 会议 3：对话管理与跨对话知识共享

时间：2026-05-18
参与者：全员
主题：确定用户如何管理对话，以及历史知识如何影响后续 Agent

### 讨论内容

- 用户会围绕同一个行程进行多轮规划，需要恢复历史 conversation。
- 如果 Agent 只记住当前一轮输入，用户每次都要重复说明偏好、同行人和禁忌。
- 如果把内部上下文摘要直接放在聊天区，会干扰普通用户理解。
- 用户必须能清除当前行程的助手记忆，避免旧上下文继续影响新规划。

### 会议结论

- 后端保存 Agent session 和 trace，前端提供历史入口恢复行程会话。
- `ConversationContextService` 负责把历史对话压缩为上下文摘要，供下一轮 Agent 使用。
- `MemoryService` 和 `/api/memories` 管理跨对话保存的偏好记忆。
- 偏好设置页提供查看、编辑和清除记忆入口；清除当前行程记忆时同步删除 session 和 trace。
- 普通助手对话只展示用户可理解的规划结果，不展示内部摘要和 trace 参数。

## 会议 4：Skill 添加和 Skill Creator 设计

时间：2026-05-28
参与者：全员
主题：确定用户如何向 Agent 添加旅行风格 Skill

### 讨论内容

- Skill 需要既能从现有内容导入，也能从当前行程和对话中沉淀。
- 直接自动发布风险较高，用户必须确认规则、禁忌和适用场景。
- 导入 Skill 是上下文操作，不应被记录为 Agent 对行程画布的“本轮改动”。
- 多个 Skill 同时使用时，需要让用户看到它们对规划的影响，而不是只显示内部工具状态。

### 会议结论

- Skill 广场提供浏览、推荐、收藏和使用旅行风格。
- Skill Creator 采用引导式流程，支持从当前行程、最近对话和外部攻略生成草稿。
- 用户确认后才能发布 Skill，发布和后续内容更新写入版本历史。
- Skill 挂载到当前行程后，Agent 读取 Skill 规则参与后续规划；本轮 diff 只展示实际画布改动。
- 右侧助手展示“当前风格”和“本轮风格影响”，使用旅行规划语言，不暴露 tool call。

## 会议 5：地图路线与 Agent 工具编排优化

时间：2026-05-29
参与者：全员
主题：补齐地点、路线、天气和时间风险闭环

### 讨论内容

- 旅行规划不是只安排景点，还需要路线、交通方式和时间可行性。
- 用户会提出“取消 A 到 B 的路线”“比较几种交通方式”“路线会晚到，先给方案”等细粒度请求。
- 在线模型可能漏调某个工具，服务端需要用明确意图做兜底。
- 地点替换后不能继续沿用旧路线，否则地图和行程会不一致。

### 会议结论

- 地图服务统一封装 POI、路线和天气。
- Agent 支持点名 POI 新增、已有活动地点替换、指定路段交通、交通方式比较、交通段取消和路线晚到修复。
- 路线晚到支持顺延下一项、缩短上一站、整体顺延后续安排和改用更快交通方式。
- 如果用户要求只看方案，Agent 只返回可选方案；用户选择方案后再写入画布。
- 地点变化会清理相邻旧路线，后续由路线补全重新计算。

- 重写需求分析、详细设计和会议记录，重点补充对话管理、跨对话知识共享和添加 Skill 的实现逻辑。
