## 用户标注的五类问题

1. 窄屏录屏下表单和布局不稳定。
2. `更新天气` 这类后台任务不应暴露给普通用户。
3. 顶部状态和操作按钮样式不一致。
4. 地图布局拥挤，地图能力没有被清晰呈现。
5. 拖动文本冗余，但景点仍需要真实可拖动排序体验。

## 额外发现的同类问题

### P1：1399px 下右侧助手变成遮罩抽屉，违背核心三栏工作台预期

用户已确认核心布局是左侧导航、中间行程画布、右侧 Agent 对话栏。当前 1399px 截图中，助手不是稳定右栏，而是点击顶部按钮后以遮罩抽屉覆盖并压暗地图。用户无法一边看清画布一边和助手对话，也无法实时感知 Agent 更新画布。

建议：
- 将常驻右栏断点从超宽屏下调到更常见的桌面宽度。
- 真正窄屏再切换为全屏助手页或底部 sheet。
- 遮罩只用于临时 modal，不用于核心协作面板。

已处理：
- 工作台三栏断点从 `2xl` 下调为 `min-[1360px]`，1399px 下右侧助手以 360px 常驻栏显示，不再通过遮罩抽屉打开。
- 1280px 仍保留两栏画布优先布局，避免回退到之前的中间画布挤压问题。
- 浏览器复查 1399px：grid 为 `260px 779px 360px`，助手 `position: static`，打开按钮隐藏，遮罩在该断点 `display: none`。证据见 `16-desktop-assistant-rail-after.png` 和 `21-desktop-assistant-rail-after-final.png`。

### P1：移动端助手覆盖画布但缺少“修改闭环”

390px 下助手占据几乎全屏，但仍露出左侧画布边缘，像临时抽屉。面板内只有当前风格、浏览风格、空白消息区和输入框，没有明确展示“本轮准备修改什么”“已改了什么”“如何回到画布确认”。

建议：
- 移动端助手改为明确的全屏协作页，顶部显示当前行程、当前日期和返回画布。
- Agent 完成后最后一条消息使用结构化 diff 块，而不是普通文本。
- 人类手动编辑继续不进入 diff；只保留 Agent 回合 diff。

已处理：
- 390px 下助手抽屉默认全宽，不再保留左侧画布缝隙；`sm` 以上仍保留右侧面板宽度。
- 助手抽屉层级提升到 `z-[1000]`，遮罩提升到 `z-[900]`，避免高德控件和版权压到助手面板之上。
- 浏览器复查 390px：助手面板 `x = 0`、`width = 390`、`right = 390`，覆盖完整视口宽度。证据见 `19-mobile-assistant-fullwidth-after.png` 和 `20-mobile-assistant-over-map-after.png`。

仍待处理：
- 移动端助手仍缺少独立的“本轮改动 / 回到画布确认”闭环，后续需要重构消息输出结构。

### P1：地图下方地点/路线概览在移动端实际横向裁切

390px 地图首屏中，地点卡第二张被裁切，只显示一部分内容。根因是地图底部列表在小屏仍尝试横向排布，同时自身还有独立滚动。用户看到的是被截断的卡片，而不是可扫描的地点列表。

建议：
- 小屏地点列表强制单列，`sm/md` 以上再启用多列。
- 地图收起态只显示当前选中地点或简短摘要，完整地点/路线列表放到可展开区域。
- 所有地点卡、路线卡加 `min-w-0` 和明确截断策略，避免撑出容器。

已处理：
- `MapOverview` 底部概览增加稳定测试钩子，收起态移动端使用 `overflow-visible`，不再把地点/路线概览变成独立滚动容器。
- 当前日期地点列表改为移动端单列，`sm` 以上才启用 `auto-fit` 多列，避免 390px 下第二张地点卡被横向裁切。
- 浏览器复查 390px：地点卡右边界均在视口内，`clippedPlaces = 0`。证据见 `13-mobile-map-overview-after-workbench.png`。

### P1：地图概览和编辑模式已分离，仍需继续细化地图交互

移动端尝试向下滚动时，手势曾被地图底部概览的内部滚动区域吃掉，主画布不动。用户必须把手指移到很窄的边缘才能继续看时间线。这和“地图要能拖动/缩放”不是矛盾，而是需要更明确的地图模式。

建议：
- 收起态地图禁用滚轮/触摸缩放，优先让页面滚动连续。
- 展开地图后进入地图编辑模式，再启用完整拖动、缩放、搜索和地点选择。
- 地图底部概览不要再作为独立滚动容器占据首屏。

已处理：
- 收起态地图概览不再有内部 `overflow-auto`，主画布滚动容器接管手势。
- 地图主按钮从 `展开地图 / 收起地图` 改为 `编辑地图 / 完成地图编辑`，明确进入可操作模式。
- 概览模式只显示地图、指标摘要和当前日期地点概览，不再默认展示搜索、地点列表和路线段列表。
- 编辑模式显示搜索框、搜索结果、地点列表和路线段列表，保留 `max-h-[42vh] overflow-auto`，避免长列表撑爆页面。
- 浏览器复查 390px：从地图概览区域滚动后主画布 `scrollTop = 650`，概览面板 `scrollTop = 0`。证据见 `14-mobile-map-scroll-after.png`。
- 浏览器复查工作台：概览模式没有搜索框、`map-day-place-list`、`map-day-route-list`，编辑模式三者均出现。证据见 `30-map-overview-mode-after.png` 和 `31-map-editing-mode-after.png`。
- 地图编辑模式搜索地点后，候选地点会进入地图预览状态，提示 `已在地图上预览 N 个地点，选择后加入 Day X`；高德候选 Marker 参与地图渲染和 fit view。证据见 `32-map-search-preview-after.png`。

### P1：路线卡维护型动作和来源说明已收口，仍需继续减少常驻操作

此前路线卡中仍有 `更新路线`，并直接显示“路线、距离和耗时来自高德”。普通用户需要的是交通方式、耗时、风险和可调整入口，不需要理解后台路线刷新动作。真实高德来源可以保留，但应降级为详情信息。

建议：
- 路线主动作改成用户语义，例如“重新规划”或“调整路线”。
- 只有路线失败、交通方式变化、地点变化后再提示需要重新规划。
- 来源信息放进“路线细节”，摘要层只保留方式、距离、耗时、时间提醒。

已处理：
- 路线来源标签改为 `实时路线 / 参考路线 / 手动调整`。
- 路线卡默认不再显示“路线、距离和耗时来自高德”。
- 单段路线动作改为 `重新规划路线 / 规划路线`。
- 浏览器复查确认工作台可见文本中没有 `高德路线`、`路线、距离和耗时来自高德`、`更新路线`。
- 地图层新增 `路线风险` 汇总，当前可见日期存在交通晚到风险时，地图区域直接显示风险数量、路线和预计到达提示；点击风险项会选中对应路线。证据见 `33-map-route-risk-summary-after.png`。
- 路线风险卡新增 `延后下一项到 HH:mm` 的快速调整入口。点击后会把下一项活动开始时间调整到预计到达时间，并按同样延迟移动结束时间；这属于手动画布编辑，不生成 Agent diff。浏览器复查证据见 `34-map-route-risk-quick-action-after.png` 和 `35-map-route-risk-quick-action-applied.png`。

### P1：主画布缺少稳定的当前日期/当前位置锚点

页面使用内部滚动容器。滚到活动和路线区后，地图、日期导航和时间线起点都可能离开视野，只剩顶部全局标题和操作按钮。用户在录屏或答辩演示中容易失去上下文：当前是在 Day 1、哪一段路线、哪个活动正在编辑。

建议：
- 日期导航做成画布内 sticky 子导航。
- 活动编辑区顶部显示当前 Day、选中活动和相邻路线摘要。
- 全局 header 保持轻量，不承担局部编辑上下文。

已处理：
- 中间画布新增 sticky Day 上下文栏，将日期切换、添加活动、当前 Day 指标、天气、路线完成数和选中对象摘要固定在画布内。
- 用户选中活动后，上下文栏显示 `正在编辑`、活动名、时间、地点和类型；用户选中路线后，上下文栏显示 `正在查看`、路线名、交通方式、路线来源和距离/耗时。
- 前端测试覆盖默认 Day 指标、选中活动摘要和选中路线摘要，避免长行程滚动后丢失当前编辑对象。
- 浏览器复查使用当前本地真实 POI 数据确认默认态、活动选中态和路线选中态均正常。证据见 `36-day-context-default-after.png`、`37-day-context-activity-after.png`、`38-day-context-route-after.png`。

### P2：活动编辑表单仍然和地图卡、时间线卡重复表达同一对象

同一个活动同时出现在地图地点卡、时间线卡、展开编辑表单中。真实 POI 场景下，编辑表单又展示地址、电话、时间预算、活动内容、提醒等字段，仍然有“表单压过行程”的感觉。

建议：
- 地图地点卡只做定位和选中，不暗示编辑。
- 时间线卡只展示摘要和排序。
- 编辑区作为稳定 inspector，默认只展示核心字段；地址、电话、营业时间放到 POI 详情折叠区。

### P2：拖拽入口已经变干净，但拖动反馈仍不完整

去掉“拖动”文本后，卡片左侧把手更专业，但用户仍不知道拖动过程中会出现什么、是否可跨天移动、拖动后路线是否会更新。当前还依赖上移/下移按钮兜底，拖拽体验没有形成闭环。

建议：
- 拖动时显示目标位置占位线。
- 支持跨 Day 拖动时给出日期区域高亮。
- 拖动完成后自动标记受影响路线为需重新规划或直接重新规划。

### P2：顶部状态、动作和工作台入口仍有层级压力

顶部同时有保存状态、助手、添加日期、导出。虽然比之前收敛，但对普通用户来说，主任务仍应是“规划行程”，不是先理解一排工具。普通工作台中的答辩评估入口已隐藏。

建议：
- 顶部主动作保留“添加日期 / 导出”，助手在桌面作为右栏存在，不需要顶部常驻按钮。
- 保存状态只做轻量反馈，不和动作按钮放同一视觉重量。
- 答辩评估保留为开发/答辩后台能力，但不在普通用户工作台渲染。

### P2：旅行风格状态太薄，无法解释“导入后如何参与规划”

助手顶部显示“当前风格 / 未选择风格 / 浏览风格”，已经避免了内部技术文案，但还不足以支撑业务亮点。用户不知道导入风格后会影响哪些内容，也不知道本轮 Agent 是否真的使用了该风格。

建议：
- 已导入风格显示为稳定状态栏，包含风格名和移除入口。
- Agent 回复中说明“本轮使用了哪些风格信号”，但不要暴露子 Agent trace。
- Skill 浏览作为独立 browse 面板，不挤在对话消息流中。

已处理：
- 助手消息渲染会识别 `本轮改动` 段落，并把 Agent diff 渲染成独立的“本轮改动”分组。
- 改动分组内显示“已应用到画布”、参考的旅行风格和结构化清单，避免 diff 混在普通段落里。
- Agent 结果消息会携带 `changeSet` 元数据，用于显示参考风格和保留撤销快照；旧字符串消息仍可兼容渲染。
- 用户可以点击“撤销本轮改动”回退 Agent 本轮写入的画布状态；撤销后 diff 块状态改为“已撤销”。
- 自动化测试覆盖“运行 Agent 后出现可访问的本轮改动分组、参考风格、清单项，并能通过持久化恢复接口撤销新增活动”。浏览器插件当前无法向 textarea 输入，动态交互以测试为准。

仍待处理：
- 单轮 Agent 撤销已通过 `POST /api/itineraries/:id/restore` 写回 SQLite；如果后续要做完整历史版本和多端撤销列表，还需要 revision history。
- diff 块还没有“回到画布确认”的定位入口。

## 建议修复顺序

1. 地图概览/编辑模式、搜索候选预览、路线风险总览、风险快速调整入口和画布 sticky Day 上下文已完成第一轮收口；后续继续做地点定位、搜索结果联动高亮和路线段选中后的画布定位。
2. 再修 1399px 常驻右栏和移动端助手全屏流程，让 Agent 协作真正闭环。
3. 路线卡语义已完成第一轮收口；后续继续减少常驻操作，把更多维护动作放入详情或异常态。
4. 然后继续重构活动编辑 inspector，减少地图、卡片、表单的重复表达。
5. 最后补完整拖拽反馈、风格状态说明和后台验收层级。
