## 2026-06-15 现场复查追加结论

复查视口：`1399x884`、`699x884`  
复查页面：`http://localhost:5173/` 当前工作台，真实高德地图已加载。

这轮复查确认，用户标注的五个问题已经不能按“单点修复”处理。它们背后还有三类同源问题：

1. **主画布被辅助面板和大块模块挤压**：在 `1399px` 桌面宽度下，布局为左侧 `260px`、中间 `779px`、右侧 `360px`。右侧助手符合三栏目标，但中间画布已经偏窄，地图、行程信息、Day 上下文栏和活动列表都在纵向堆叠。后续不能再向中间主流里追加大块表单或说明。
2. **编辑模式把工具放出来了，但没有建立“编辑工作区”**：地图编辑态显示搜索和真实地图，但地点列表、路线段列表落在首屏下方，尤其 `699px` 窄屏时首屏几乎只剩搜索框和地图。用户会感觉能搜索和看地图，但不知道在哪里管理已选地点和路线。
3. **维护动作需要降级到选中态/详情态**：路线卡和活动卡里的维护动作都是必要能力，但默认态不应和地点标题、时间、路线风险抢同一视觉层级。本轮已先把活动排序/删除、路线交通方式/重新规划从常规摘要层收进选中态或详情态，后续继续补拖拽占位和更完整 inspector。

### P1：地图编辑态首屏失衡

当前 `编辑地图` 后，地图区域高度为 `68vh`，搜索框在地图上方，地点/路线列表在地图下方。桌面端列表刚好落在首屏底部，窄屏端完全不可见。

影响：
- 用户不知道“已选地点”和“路线段”可点选、可联动画布。
- 搜索结果、已选地点、路线段没有形成一个连续的编辑流程。
- 录屏中会看起来像只做了地图嵌入，而不是完整地图编辑。

建议：
- 编辑态改为“地图 + 右侧/底部编辑栏”的工作区：搜索、候选、已选地点、路线段至少有一个区域始终可见。
- 窄屏下地图高度控制在 `45-52vh`，把已选地点/路线段放到地图下方的 tab 或 sticky 抽屉。
- 概览态继续保持轻量，不显示搜索和长列表。

已处理：
- 地图卡在编辑态标记为 `地图编辑工作区`，便于测试和辅助技术识别当前区域。
- 编辑态顺序调整为：搜索栏 -> 已选地点/路线段 -> 地图视口；地点和路线段不再被地图压到首屏下方。
- 编辑态地图高度从 `68vh` 降为窄屏约 `45vh`、中大屏约 `50vh`。
- 概览态删除“进入地图编辑后可搜索地点、选择地点卡和查看路线段。”这类教学说明句。
- 前端测试新增覆盖：地点列表和路线段列表在可编辑地图视口之前；概览态不显示教学说明句。
- 浏览器复查：`1399x884` 下地点列表 `y=311`、路线段 `y=283`、地图 `y=415`；`699x884` 下地点列表 `y=352`、路线段 `y=452`、地图 `y=584`，地点和路线段均在首屏可见。证据见 `39-map-edit-workspace-after.png`、`40-narrow-map-edit-workspace-after.png`。

### P1：行程信息表单位置仍打断主流程

`编辑信息` 展开后，基础信息表单插在地图和 Day 时间线之间，并把 Day 导航和活动列表挤到首屏外。它虽然比早期输入框堆叠更整齐，但仍属于“表单压过画布”。

影响：
- 用户修改预算或日期时无法同时观察地图和当天安排。
- 窄屏录屏中会出现大块表单，使产品重点从“规划画布”变成“填表”。

建议：
- 行程信息改为右侧 inspector、顶部弹层或轻量抽屉。
- 主画布只保留标题摘要和“编辑”入口。
- 展开态字段按“基础信息 / 预算与偏好 / 备注”分组，默认只显示基础字段。

已处理：
- 主画布只保留 `行程信息` 摘要卡和 `编辑信息` 入口，`返回日期`、`行程备注` 等字段不再出现在摘要卡内。
- 点击 `编辑信息` 后打开 `编辑行程信息` 弹窗；字段集中在弹窗内完成编辑，保存按钮为 `应用信息`，保存后关闭弹窗。
- 弹窗层级提升为 `z-[1200]`，高于高德地图控件、版权层和移动端助手，避免第三方地图元素穿透表单。
- 前端测试新增覆盖：行程信息表单在 dialog 内，摘要卡不包含 `返回日期`，并校验弹窗外层为 `z-[1200]`。
- 浏览器复查：`1399x884` 下摘要卡 `top=603/bottom=680`，Day 上下文栏 `top=700`，弹窗打开后 `返回日期` 和 `行程备注` 只存在于 dialog；`699x884` 下弹窗 `left=16/right=683`，地图版权和缩放控件不再压到表单上。证据见 `41-trip-info-dialog-desktop-after.png`、`42-trip-info-dialog-narrow-after.png`。

### P1：第三方地图控件可能穿透应用级弹窗

在行程信息弹窗改造后的首次浏览器复查中，高德地图版权和缩放控件曾显示在弹窗输入框上方。根因是应用弹窗仍使用通用 `z-50`，低于地图 SDK 内部控件层级。这和此前“助手抽屉被地图控件压住”属于同类问题：第三方地图不是普通背景内容，它有自己的 overlay 层，必须纳入产品视觉栈管理。

影响：
- 用户会看到地图版权、缩放控件或 Marker 标签压在表单、抽屉、确认弹窗上，显得不专业。
- 窄屏录屏时尤其明显，因为弹窗接近地图顶部，地图控件更容易与输入框重叠。
- 后续新增 Skill 导入、导出确认、删除确认等弹窗时也可能复发。

建议：
- 建立统一 z-index 层级：普通画布 < 地图 SDK < sticky 子导航 < 助手抽屉 < 全局 modal。
- 所有全局 modal 使用高于助手和地图控件的层级；地图内业务浮层只在地图容器内部管理。
- 浏览器验证时增加 `elementFromPoint` 或截图检查，确认弹窗打开后顶层元素属于弹窗，而不是地图控件。

已处理：
- `编辑行程信息` 弹窗和 `导入旅行风格` 弹窗外层从 `z-50` 提升到 `z-[1200]`。
- 浏览器复查桌面与窄屏：弹窗区域顶层元素已为弹窗遮罩或输入框，高德控件不再穿透。证据见 `41-trip-info-dialog-desktop-after.png`、`42-trip-info-dialog-narrow-after.png`。

### P1：右侧助手空态和风格浏览仍占用过多协作空间

右侧助手常驻后是正确方向，但当前空态消息少、对话区留白大；打开“浏览风格”后，风格列表占据右栏上半部分，聊天输入仍在底部，用户不容易判断当前是在选择风格还是继续对话。

影响：
- “导入风格参与规划”的亮点没有被清晰证明。
- 风格浏览和对话共用一条纵向流，容易挤压 Agent diff 和确认路径。

建议：
- 顶部保留固定风格状态条；“浏览风格”打开独立 browse 面板，并提供关闭/应用后的明确反馈。
- 空态增加可执行建议，但只保留 2-3 条，不写技术说明。
- Agent 每轮最后输出结构化 diff，显示本轮使用的风格信号、改动清单、定位和撤销。

已处理：
- 右侧助手只保留固定的 `当前风格` 状态条；风格列表不再内联占用对话区上半部分。
- `浏览风格` 打开 `z-[1200]` 的全局右侧选择面板，面板作为助手 rail 的兄弟节点渲染，不在助手消息流内部。
- 应用风格后自动关闭选择面板，只更新风格状态和导入消息，不生成 `本轮改动`；Agent diff 仍只用于 Agent 轮次。
- 浏览器复查：打开面板时 `inlineChooserInAssistant=false`，消息区高度不被压缩；应用风格后面板关闭，状态条显示已导入风格。证据见 `47-skill-browser-panel-after.png`、`48-skill-browser-applied-after.png`。
- 前端测试覆盖选择面板层级、宽度、关闭行为，以及导入风格不生成 Agent diff。

### P1：活动和路线卡默认态仍过重

活动卡默认显示拖拽、序号、类型、地点、时间、编辑、上移、下移、删除；路线卡默认显示路线标题、来源、距离耗时、摘要、交通方式、路线细节、重新规划。功能完整，但阅读层级不够清晰。

影响：
- 用户扫描一天行程时会被工具按钮打断。
- 长行程下页面高度增长很快，移动端尤其明显。
- “拖动景点”虽然已有手柄，但缺少拖动占位和落点反馈。

建议：
- 活动卡默认只展示：时间、标题、地点、类型、关键提醒。
- 编辑、删除、上移、下移收进选中态或更多菜单；拖拽手柄保留为纯图标。
- 路线卡默认只展示：交通方式、耗时/距离、风险；重新规划只在异常、方式变更或详情态出现。
- 拖动时显示落点占位线，完成后自动更新受影响路线状态。

已处理：
- 活动卡默认态只保留拖拽把手、站点序号、标题、类型、地点、时间/预算和一个 `编辑/补全` 入口；`上移`、`下移`、`删除` 收到选中态，避免长行程扫描时被工具按钮打断。
- 选中活动后仍会打开完整活动编辑区，并在对应卡片上显示排序和删除操作，纯手动规划能力保留。
- 路线卡默认态不再常驻交通方式下拉和 `重新规划`；常规路线只显示路线摘要、状态、距离/耗时和 `路线细节`。
- 打开路线详情后才显示交通方式选择和 `重新规划`；路线缺失或失败时仍保留必要的快速 `规划路线/重新规划` 入口。
- 拖拽活动时，原卡进入半透明拖动状态，当前可投放目标卡显示 `data-drop-target=true` 和顶部主色落点条；拖到自身不会显示落点，放下或结束拖拽后状态清空。
- 前端测试覆盖：默认态不出现 `上移湖滨咖啡`、`删除湖滨咖啡`、路线交通方式下拉和 `重新规划`；选中活动或打开路线详情后这些手动能力可用。
- 前端测试同步覆盖拖拽中间状态：拖动 `湖滨咖啡` 时 `activity-drop-1` 标记为 dragging，悬停 `activity-drop-0` 后显示 `activity-drop-indicator-0`，drop 后重排且落点条消失。
- 浏览器复查：默认态 `hasReplan=false`、`hasRouteModeSelect=false`、`hasMoveUpCafe=false`、`hasDeleteCafe=false`；选中活动后通过 aria-label 确认 `上移/删除` 可用；路线详情打开后 `hasModeSelect=true`、`hasReplan=true`。证据见 `43-card-density-default-after.png`、`44-card-density-activity-selected-after.png`、`45-card-density-route-details-after.png`。

### P2：主画布仍有解释性文案

例如地图概览底部的“进入地图编辑后可搜索地点、选择地点卡和查看路线段。”属于产品自述，不是用户当前需要的信息。类似文案应减少，改成可执行控件或空态提示。

建议：
- 有数据时不显示教学文案。
- 空态只写下一步动作，例如“搜索地点加入 Day 1”。
- 能用按钮/状态表达的，不用说明句表达。

## 总体判断

用户标出的五个点不是孤立问题，而是同一类设计债的不同表现：当前界面仍然把“内部系统能力”和“用户规划任务”混在同一个层级里。旅行用户需要看到的是目的地、日期、地图、地点、路线、预算和可编辑日程；不应该先理解天气刷新、路线刷新、Agent 机制、数据来源、调试入口、重复表单和技术状态。

后续改动应围绕一个原则：普通用户界面只呈现“规划对象”和“可执行决策”，维护动作自动化或降级到详情/后台。

## P1 问题

### 1. 工作台入口不可恢复，刷新后回到首页

桌面和移动端刷新后会回到产品首页，需要再次点击“进入工作台”。这会影响答辩演示，也不符合真实规划产品的使用习惯。一个行程应该有稳定 URL 或至少有稳定的本地恢复入口。

建议：
- 为当前行程提供稳定工作台路由，例如 `/trips/:id`。
- 首页只负责介绍、新建和进入最近行程。
- 刷新后应恢复最近打开的行程画布，而不是回到介绍页。

已处理：
- 应用新增轻量 hash route：`#/workbench`、`#/skills`、`#/creator`、`#/settings`、`#/evaluation`，不引入额外路由库。
- 所有主导航入口和内部跳转都会同步 URL；浏览器前进/后退通过 `hashchange`/`popstate` 恢复对应页面。
- 点击首页 `进入工作台` 后 URL 从 `/` 变为 `#/workbench`；刷新后应用直接初始化到工作台，不再回到首页。
- 前端测试新增覆盖：进入工作台后 hash 为 `#/workbench`，重新挂载应用后仍显示 `行程地图`，且首页 `进入工作台` 按钮不再出现。
- 浏览器复查：起始页 `hash=""` 且显示 `进入工作台`；进入后 `hash="#/workbench"`；刷新后仍为 `hash="#/workbench"`，显示 `行程地图` 且不显示首页入口。证据见 `46-workbench-route-restore-after.png`。

### 2. 地图仍然是“展示地图”，不是“地图编辑模式”

当前已经有真实高德地图、搜索框和“展开地图”，但用户仍难以判断地图能做什么。地图上路线标签、地点标签、缩放控件、版权、搜索、地点列表和路线列表集中在一个区域，视觉上拥挤；“展开地图”也没有清晰表达展开后会进入地点搜索、选点、拖动、路线查看还是全屏浏览。

建议：
- 把地图分为“概览模式”和“编辑模式”。
- 概览模式只保留路线轮廓、当天地点摘要和一个明确的“在地图中编辑”入口。
- 编辑模式提供搜索结果、添加到当天、替换地点、调整顺序、查看路线详情。
- 地图版权和缩放控件保留，但不要和业务信息争夺底部空间。

已处理：
- 默认地图进入概览模式，只保留路线轮廓、地图指标摘要和当前日期地点概览。

## P2 问题

### 8. “旅行风格 Skill”状态还不足以解释业务亮点

右侧助手顶部显示“当前风格 / 未选择风格 / 浏览风格”，比之前的内联列表更干净，但还没有形成业务闭环。用户不知道导入风格后会影响哪些规划决策，也不知道 Agent 本轮是否真的使用了风格。

建议：
- 助手顶部保留一个固定风格状态条，只显示已导入风格和移除入口。
- 浏览风格打开独立 browse 面板，不塞在对话流里。
- Agent 最后一条输出的 diff 中显示“参考风格”，不要暴露子 Agent trace。
- 创作 Skill 和从行程总结 Skill 应有独立页面或弹窗闭环。

### 9. Agent 对话区空态过于简单

当前空态只告诉用户“告诉我想调整的日期、地点、节奏或预算”。它没有解释本轮会直接修改画布，也没有给用户确认修改的路径。对于一个会写入画布的 Agent，空态需要更像协作面板，而不是普通聊天框。

建议：
- 空态提供 2-3 个高质量例句，覆盖补地点、改节奏、降预算。
- 明确“发送后会更新画布，改动会在下一条消息中列出”。
- Agent diff 作为最后一条输出继续保留，附带撤销和定位到画布入口。

### 10. 顶部按钮和状态的层级仍然接近

“已保存”“添加日期”“导出”在顶部同一视觉强度附近。保存状态是反馈，不是操作；添加日期是规划动作；导出是结果动作。当前按钮样式虽然比之前一致，但语义层级还不够清楚。

建议：
- 保存状态降级为轻量文本或小点状态。
- 添加日期作为次主操作。
- 导出放入结果/更多菜单，或在行程完成度较高时强化。

### 11. 重复摘要占用主画布空间

标题区、地图摘要、行程信息卡、Day 标题都在重复目的地、日期、天数、预算等信息。重复本身不是问题，但当前重复没有服务于上下文，而是在首屏增加了视觉噪音。

建议：
- 全局 header 显示行程标题、目的地、日期。
- 地图卡只显示地图相关摘要。
- 行程信息卡改为 inspector 或可折叠详情，不长期占据主流。

已处理：
- 本轮先删除全局 header 内重复的旅行风格计数，把风格上下文收敛到助手 rail；标题区只保留行程标题、目的地、日期、保存状态和高频结果动作。
- 后续仍可继续把 `行程信息` 摘要并入更轻的 inspector，但这轮不再继续压缩主画布结构，避免影响手动编辑闭环。

### 12. 文案命名混用“Skill / 风格 / 助手 / Agent”

业务对用户应讲“旅行风格”或“风格 Skill”；技术实现和文档里可以讲 Agent、Skill。当前界面仍有较多英文 Skill，但用户侧不需要理解 Codex Skill 标准。

建议：
- 用户侧主文案使用“旅行风格”。
- 在创建/导入场景中补充“Skill 格式”作为格式说明。
- 技术术语放到开发文档、答辩后台和 PPT 技术页。

已处理：
- 保留用户已指定的一级入口 `创作 Skill`、`Skill 广场`，本轮不擅自改名。
- 导出预览中的课程/答辩语境已从普通工作台移除，文案由 `完整行程 Markdown，可用于答辩或分享。` 改为 `完整行程 Markdown，可用于分享或归档。`
- 前端测试新增覆盖：普通工作台不显示 `当前风格 N 个` 这类重复全局状态，也不显示 `答辩或分享` 的导出说明。

### 13. 可访问性和视觉细节需要收口

当前 icon-only 控件大多有 aria label，这是好的；但视觉上仍有一些细节影响专业感，例如温度显示为 `22-28 C`、地图标签被路线和底部控件压住、移动端按钮行容易显得拥挤。

建议：
- 温度显示为 `22-28°C`。
- 长地点名、路线名统一使用两行截断或详情展开。
- 移动端主操作按钮保持 44px 以上触控高度，并减少同屏按钮数量。

已处理：
- 画布内天气温度展示会把 `22-28 C` 这类原始值格式化为 `22-28°C`，保留底层天气数据结构和导出逻辑不变。

### 14. 移动端首屏仍被全局 header 和地图吞掉

在 390x844 下，header 高度约 162px，地图卡约 474px，Day 上下文只在首屏底部露出一条，活动列表完全不可见。对录屏或窄屏演示来说，用户第一眼看到的是标题、按钮和地图，而不是“我正在编辑 Day 1 的行程”。

建议：
- 移动端 header 压缩为标题一行 + 次级状态一行；`助手/添加日期/导出` 改为 icon 或更多菜单。
- 移动端地图默认进入更短摘要态，高度控制在 280-340px；需要编辑时再进入全屏/半屏地图工作区。
- Day 上下文应在移动端更靠前，作为“当前编辑锚点”，活动列表至少露出第一张卡片顶部。

优先级：P1。它直接影响“窄屏视频/答辩演示”中用户是否能看懂产品核心功能。

### 15. 左侧会话记录缺少去重和可识别性

桌面复查中，左侧会话记录出现多个 `杭州周末旅行 杭州 / 3 天`，其中只有当前行程有状态前缀。虽然多行程切换功能已经存在，但重复标题会让用户难以判断自己要打开哪一个。

影响：
- 用户可能误点同名旧行程，产生“数据被覆盖/回退”的错觉。
- 会话记录看起来像测试数据堆积，降低产品完成度。
- `当前：高德骑行验证行程` 这种前缀在视觉上占用主文本，但不是最有效的识别方式。

建议：
- 会话记录增加更新时间、日期范围或最近编辑摘要。
- 同名行程自动显示目的地/日期差异；完全重复时显示 `副本` 或创建时间。
- 当前行程用左侧竖条、选中背景或小状态点表达，不把 `当前：` 写进主标题。

优先级：P1。它属于“状态反馈和内容标题混写”的同类问题，也会影响多行程闭环。

### 16. 用户侧仍有技术词和供应商词泄漏

当前普通工作台仍能看到 `创作 Skill`、`Skill 广场`、行程名中的 `高德骑行验证行程`、地图内 `AutoNavi` 版权和路线来源上下文。地图版权不可删除，但业务内容中不应出现验证/供应商测试语气；`Skill` 作为一级入口是用户暂定决策，但页面内部应尽量解释为“旅行风格”。

影响：
- 用户会把产品理解成技术演示，而不是旅行规划工具。
- 供应商/测试词进入标题和列表，会污染截图、PPT 和答辩录屏。
- 业务亮点“风格分享平台”没有被自然表达，反而显得像开发概念。

建议：
- Seed/示例行程名不使用 `高德验证`、`测试`、`demo` 等词，改成真实旅行命名。
- 用户侧主体文案统一讲“旅行风格”，只有文件导入、创作页辅助说明里保留 `Skill`。
- 地图版权保留；路线卡和标题不把供应商作为主信息，只在详情或来源说明出现。

优先级：P1。它和用户标出的“高德地图已加载”“Agent 输入”等同源，都是内部实现泄漏到普通用户界面。

### 17. 顶部结果动作仍没有形成稳定动作模型

桌面和移动端都显示 `已保存 08:11 / 添加日期 / 导出`，窄屏还额外出现 `助手`。这些都是必要能力，但保存状态、规划动作、结果动作混在一排，按钮形态接近。

影响：
- 用户不清楚当前最重要的下一步是什么。
- 移动端 header 被动作区撑高，进一步压缩画布。
- 后续如果继续增加分享、复制、导入等动作，header 会很快失控。

建议：
- 保存状态降级为标题旁状态点或短文本。
- `添加日期` 留在 Day 上下文附近，而不是全局 header。
- `导出` 进入更多菜单或结果面板；当行程完整度较高时再强调。
- 移动端只保留一个主入口和一个更多菜单。

优先级：P1。它是用户标出“按钮样式不一致”之后仍未完全解决的同类语义层级问题。

### 18. 助手空态仍像说明卡，不像可确认的协作流程

移动端助手全屏态中，空白区域很大，主要文案是 `告诉我想调整的日期、地点、节奏或预算，我会直接更新行程。`。它说明了能力，但没有给出“会改哪里、如何确认、如何撤销、是否参考当前风格”的路径。

影响：
- 用户可能担心发送后直接覆盖画布。
- 已导入风格只作为顶部状态出现，没有在对话流程中形成闭环。
- 移动端全屏助手缺少当前行程和返回画布后的确认锚点。

建议：
- 空态改为 2-3 个可点选的任务建议，例如补 Day 2 室内活动、降低交通切换、压缩预算。
- 在输入区上方固定提示：本轮修改会写入画布，下一条消息列出改动并可撤销。
- Agent diff 中显示 `参考风格`、改动位置和撤销入口；不要把人类手动编辑计入 diff。
- 移动端助手顶部显示当前行程名和返回画布入口。

优先级：P1。它直接关系到 Agent 写画布的信任感。

### 19. 地图与时间线缺少清晰的“主从关系”

桌面首屏中地图是视觉主角，Day 上下文和活动列表在地图之后；移动端更明显。用户当前反馈“地图是否只读、能否展开、拖动、缩放、搜索地点”背后的同类问题，是地图既像主视图又像辅助视图，但没有明确当前处于概览、编辑还是路线检查任务。

影响：
- 用户不知道该在地图上操作，还是在活动卡上操作。
- 地图控件、路线标签、地图摘要和 Day 摘要争抢同一层级。
- 后续加入景点搜索、路线修复、拖动排序后，如果没有模式边界，会重新变成控件堆叠。

建议：
- 概览态：地图只承担路线和空间关系预览，控件少，活动时间线是主编辑入口。
- 编辑地图态：搜索、候选地点、已选地点、路线段组成独立工作区，地图成为编辑画布。
- 路线检查态：突出异常和修复动作，隐藏普通来源/维护信息。
- 移动端地图编辑建议独立全屏，避免和时间线同时抢首屏。

优先级：P2。当前已有模式分离基础，但还需要进一步做视觉和任务边界收口。

## 推荐修复顺序

1. 先收口用户侧泄漏：示例数据去掉验证/测试语气，降级供应商和技术词，统一“旅行风格”文案。
2. 压缩移动端首屏：header 减高、地图摘要态减高、Day 上下文提前，确保第一张活动卡露出。
3. 重做顶部动作模型：保存是状态，添加日期回到 Day 上下文，导出进结果/更多菜单。
4. 完善 Agent 协作闭环：空态建议、发送后写画布说明、diff 定位、持久化撤销、参考风格说明。
