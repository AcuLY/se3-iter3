## 同类问题

### P1：助手抽屉在常见录屏宽度遮挡主画布

1399px 和 1024px 下，旅行助手以遮罩抽屉覆盖地图和表单。用户无法一边看地图/日程，一边对话或导入风格，这和“IDE 右侧对话栏”的目标冲突。

修复方向：
- 1280px 以上恢复固定右侧栏。
- 1280px 以下才使用抽屉。
- 抽屉打开时避免让底层画布看起来仍可编辑。

### P1：普通用户入口混入开发/管理能力

左侧导航把“评估后台”和“新建行程、创作 Skill、Skill 广场”放在同一功能层级。Skill 广场也把标签编辑、Skill.md 粘贴导入长期暴露在浏览页主流程里。

修复方向：
- 普通功能入口保留在上方。
- 答辩/开发后台移动到底部独立区域。
- Skill 广场默认做“浏览/推荐/收藏/用于当前行程”。
- Skill.md 导入和标签编辑改成按需展开或弹出，而不是卡片默认输入框。

### P1：编辑型表单仍然过早占据主画布

行程信息展开后，大量输入框直接推开地图和日期内容；Skill 广场卡片默认显示标签输入框；助手风格浏览列表默认显示规则和禁用项。这些都和用户之前指出的“输入框堆叠、缺少主次和分组”属于同类问题。

修复方向：
- 行程信息继续保留折叠摘要，展开后用更明确的分组和紧凑布局。
- Skill 广场卡片只显示浏览所需信息，编辑入口隐藏到二级操作。
- 助手内风格浏览改成轻量选择面板，详细规则放进“查看详情”。

### P2：空状态和提示文案重复，像说明书而不是工具

地图区域同时出现搜索框、地图内提示、底部提示。Day 1 空行程时，主画布缺少可直接开始的建议操作，默认状态看起来像空壳。

修复方向：
- 地图只保留一个空状态提示，优先放在地图画布内。
- 空日程提供“搜索地点 / 添加活动 / 使用示例地点”这类直接操作。
- 删除重复解释性句子。

### P2：Skill 浏览面板信息密度和视觉权重不对

助手内风格浏览把每个 Skill 的规则、禁用项、导入动作都塞进按钮里，红色“避免”在视觉上抢过了选择动作。用户的核心任务是快速选择风格，不是阅读 Skill 规则全文。

修复方向：
- 每个 Skill 只展示名称、短描述、2-3 个标签和使用状态。
- 规则/禁用项放详情页或折叠区。
- 导入按钮文案改为“使用”，已导入状态提供明确的移除操作。

### P2：地图的可交互能力存在，但视觉上没有被组织成“规划工具”

地图已经有缩放控件、搜索框和展开能力，但缺少“当前地图会改变什么”的明确反馈。空状态时地点列表/路线列表隐藏，用户很难预期搜索结果添加后会怎样进入日程。

修复方向：
- 搜索结果区与地图保持更近的空间关系。
- 地点加入后，在地图下方用“路线/地点”列表呈现可选对象。
- 地图摘要 chip 减少数量，避免长句。

### P3：按钮和状态 chip 的语义层级仍需统一

顶部“已保存、助手、添加日期、导出”视觉上接近同一层级；部分状态 chip 像按钮，部分按钮像状态。用户需要快速识别可点击动作和只读状态。

修复方向：
- 只读状态使用更弱的背景和无边框样式。
- 主操作和次操作的按钮样式固定。
- 图标按钮保留 tooltip/aria，文字按钮只用于明确命令。

## 当前实现复查：五个标注点之外的同类问题

### P1：1280px 固定三栏仍会挤压主编辑画布

当前断点在 1280px 恢复右侧助手栏，但左侧导航、右侧助手和主画布同时存在时，中间行程编辑区只剩约 700px。活动编辑表单在 1280px 下出现标签竖排换行，1024px 下虽然助手变成抽屉，但表单仍按桌面分栏硬压。

修复方向：
- 固定右侧助手栏的断点应上移到更宽的桌面宽度，或在 1280-1399px 之间折叠左侧导航。
- 活动编辑、路线细节、地图下方列表都应按容器宽度而不是全局断点决定列数。
- 录屏常用宽度下优先保证主画布完整，不为了常驻助手牺牲行程编辑可读性。

### P1：地图、地点列表、日程卡和编辑表单在表达同一个对象

添加两个活动后，同一个活动同时出现在地图下方地点卡、日程卡、展开编辑表单中；路线也同时出现在地图下方路线段和日程中的路线卡。用户很难判断应该在地图区点选、在日程区编辑，还是在路线卡计算。

修复方向：
- 保留一个主编辑焦点：点选地图/地点卡只负责定位和选中，具体编辑回到日程卡或专用编辑面板。
- 地图下方列表只做概览和定位，不承载复杂编辑动作。
- 选中活动后，地图和日程应同步高亮，但不要让多个区域同时看起来都可编辑。

### P1：活动编辑和路线细节可同时展开，页面焦点失控

当前可同时展开活动表单和其后的路线细节。首屏会同时出现活动基础信息、地点搜索、时间预算、活动内容、路线数据、路线调整、保存按钮。用户正在编辑“活动”还是“这段交通”不够明确。

修复方向：
- 同一天内采用单一编辑焦点：展开活动时收起路线细节，展开路线时收起活动详情。
- 路线细节可作为紧邻路线卡的二级面板，但默认只展示摘要、来源、耗时和主要动作。
- 长表单改为“摘要态 + 编辑态”，不要让每个活动默认都能无限展开。

### P2：默认新增活动质量过低，导致路线和地图信息不可读

手动添加活动默认生成“新的活动 / 杭州 / 自由活动 / 14:00-15:00”。添加两个后路线标题变成“新的活动 到 新的活动”，地图地点卡都显示“杭州”，这会让用户误以为地图没有真正关联地点。

修复方向：
- “添加活动”应先创建空白编辑态，标题用“未命名活动”弱提示，地点为空并聚焦地点搜索。
- 如果从地图搜索结果添加，则直接使用 POI 名称、地址、类型和坐标。
- 当活动名称重复时，路线标题优先展示地点名或序号，例如“01 到 02”加地点副标题。

### P2：路线计算入口重复，批量和单段关系不清

地图顶部有“批量路线交通方式 + 补全路线”，每个路线卡又有交通方式选择和“计算路线”。两个入口都合理，但当前层级接近，用户不知道应该先补全全部还是逐段计算。

修复方向：
- 地图区只保留“还有 N 段未计算，补全全部”这种状态型入口。
- 单段路线卡保留逐段交通方式和重新计算。
- 批量入口使用次级样式，并明确会影响所有待计算路线。

### P2：字段单位和标签处理不稳定

路线细节里“距离 km”“耗时”“费用”分别使用标签和 placeholder 表达单位；窄屏下活动表单标签会折行成窄列。表单可用，但扫读成本高。

修复方向：
- 单位进入输入框后缀或字段旁固定文本，例如“距离 / km”“耗时 / 分钟”“费用 / 元”。
- 主画布小于 800px 时表单改为单列或两列，不继续使用桌面三列。
- 对短标签设置最小宽度，避免“活动名称”被压成两行竖排。

### P2：Skill 浏览仍挤在对话流里，不像独立 browse 面板

助手内“浏览风格”已简化卡片内容，但打开后会占用右侧对话栏大部分高度，把对话历史和输入区挤到下方。它更像一个临时选择器，不应该长期插在对话流里。

修复方向：
- 顶部只保留已应用 Skill 状态栏和“使用风格”入口。
- 浏览风格用独立 sheet/popover/command 面板，选中后关闭并回到对话。
- 详细 Skill 信息跳到 Skill 广场或详情页，不在对话栏长列表展开。

### P3：保存语义和自动保存状态混杂

顶部显示“已保存”，行程信息、路线细节里又有“保存信息 / 保存调整”。这会让用户不确定普通编辑是否已经自动保存、哪些修改必须点按钮才生效。

修复方向：
- 如果采用自动保存，局部按钮改成“应用到路线”“完成编辑”，顶部只显示同步状态。
- 如果某些字段需要显式确认，需要在局部说明“未应用”的状态，而不是统一叫保存。
- 手动编辑不计入 Agent diff，但仍应有清楚的本地修改反馈。

### P3：选中态和主品牌红色过近，容易被误读成错误

活动卡选中态和主按钮都使用较强的红/粉色。复杂表单展开时整块红色描边容易像校验错误，而不是“当前选中”。

修复方向：
- 选中态改用中性深色、蓝色或高德地图相关的定位色。
- 红色只用于主 CTA 或危险操作，不同时承担选中、强调和提交。
- 删除按钮保持 ghost/icon 样式，但 hover/active 才显示危险色。

## 下一轮 UI 收敛建议

1. 先改断点和画布宽度：1280-1399px 不强行三栏，活动/路线表单按容器宽度降列。
2. 再改编辑焦点：活动详情和路线详情互斥展开，地图列表只做定位概览。
3. 再改新增活动流程：默认新增进入空白编辑态，地图添加使用 POI 真实信息，重复标题时路线使用地点/序号。
4. 再改路线入口：地图区保留批量补全状态，路线卡负责单段计算和调整。
5. 最后统一保存语义、选中态颜色、字段单位和 Skill 浏览弹层。

## 本轮已处理补充

- 断点和画布宽度：工作台右侧助手到 `2xl` 才常驻，1280-1399px 回到“主画布 + 助手抽屉”，避免中间行程编辑区被三栏挤压。
- 编辑焦点：点击路线细节会清空活动编辑焦点，活动详情和路线详情不会同时展开。
- Skill 浏览：助手里的“浏览风格”改成临时选择浮层，使用风格后关闭，移出风格后保留浮层并把卡片切回“使用”。
- 交通路线：路线服务、API、Agent 工具、前端 fallback 和导出都开始保留结构化路线步骤；路线详情面板展示“路线步骤”。
- 默认数据治理：手动“添加活动”不再生成“新的活动 / 杭州 / 14:00-15:00”，而是创建空白活动并进入编辑态；地图点位只接收真实地点或用户填写地点，不再把目的地当作活动地点。
- Agent fallback 治理：没有真实 POI 或工具返回时，Agent 建议活动不再把目的地写成 `placeName`，避免没有具体地点的建议直接上地图。
- 路线端点治理：批量补全和单段路线都只处理有明确起终点的活动；两个空白活动不会显示地图点、批量补全入口或路线计算卡，API 与 Agent 工具也会跳过无端点路线。
- 显示名降级：路线、地图卡、可访问标签和本地 fallback 路线步骤统一使用“活动标题 -> 地点名 -> 第 N 站 / 未命名活动”的顺序，避免空标题导致“路线： 到 ...”。
- 旧数据兼容：本地 SQLite 已存在的旧默认占位活动会在客户端进入状态前规范化为空白活动，并清理这些占位活动产生的相邻路线，避免刷新后旧假数据继续出现在地图里。

## 第六批同类问题：编辑焦点、状态语义和空状态闭环

### P1：活动卡片继续承载详情表单，时间线仍容易变成输入框堆叠

第五批修复后，新增活动已不再伪造“新的活动 / 杭州 / 14:00-15:00”，但被选中的活动详情仍在对应日程卡内展开。用户连续添加空活动时，时间线会在摘要卡、路线卡和表单之间来回穿插，视觉上仍像“表单栈”，而不是可扫读的行程日志。

修复方向：
- 时间线卡片只承载活动摘要、排序、选择和删除。
- 具体字段集中到单一“编辑活动”面板，同一时刻只服务当前选中的活动。
- 选中地图地点、日程卡或新增活动时都进入同一个编辑焦点，不让多块区域同时看起来可编辑。

已处理：
- `ActivityDetailsPanel` 作为独立编辑面板渲染在当天标题下方。
- `ActivityEditor` 改为摘要卡，不再在 `listitem` 内渲染活动名称、地点、时间、预算和备注输入框。
- 新增回归测试验证“第 3 项活动名称”只出现在 `编辑活动` region 中，不出现在时间线卡片内部。

### P2：地图空状态与状态 chip 复读同一句话

地图标题区已经用 chip 显示“Day N 还没有地点”，空状态浮层又重复“Day N 还没有地点”。这类重复和用户之前指出的无意义说明文字同源：界面在解释状态，而不是推动下一步。

修复方向：
- 顶部 chip 保留当前范围和状态。
- 地图画布空状态只给短标题和下一步动作。
- 空状态要能直接进入完整手动规划流程，而不强迫用户先找 Agent。

已处理：
- 地图空状态标题改为“还没有地点”，避免与 chip 重复。
- 空状态加入“添加空活动”按钮，与主画布“添加活动”走同一套空白编辑流程。

### P2：局部确认按钮仍使用“保存”语义，和顶部同步状态冲突

顶部持续显示“已保存 / 正在保存”，局部表单再出现“保存信息”“保存调整”会让用户误判哪些字段自动保存、哪些需要另存。当前语义更准确地说是“把当前表单值应用到行程/路线”。

修复方向：
- 保留顶部同步状态作为全局保存反馈。
- 局部按钮改成“应用信息”“应用调整”。
- 后续如果引入未应用状态，再在局部显示具体差异，而不是泛化成保存。

已处理：
- 行程信息确认按钮改为“应用信息”。
- 路线手动覆盖按钮改为“应用调整”。
- 前端测试同步改为检查新的按钮语义。

### P2：选中态仍使用品牌红色，容易被理解成错误或危险

Pinterest 红适合主 CTA，但复杂工具里红色描边/路线高亮容易被误读为错误、冲突或危险。选中态应该更像定位和焦点，而不是警告。

修复方向：
- 选中态统一使用 `ring` 定位色。
- 地图路线选中 polyline 使用蓝色焦点色。
- 红色继续保留给品牌主操作或危险操作。

已处理：
- 活动卡、地图地点卡、路线卡、路线详情卡的选中 ring 改为 `ring-ring`。
- 高德 polyline 选中颜色改为 `#435ee5`。

## 第七批同类问题：草稿安排、风格选择和浏览页管理入口

### P1：空白手动安排和地图地点没有视觉分层

用户可以手动添加空白安排，这是完整手动规划闭环必须保留的能力。但当前页面把多个空白安排显示成“未命名活动 / 未设置地点 / 未设置时间 / 未设置预算”，同时地图显示“还没有地点”。这会让用户误以为活动、地点、地图三者不同步，而不是理解为空白草稿还缺地点。

修复方向：
- 空白安排显示为“待补全安排”，不要用“未命名活动”这种占位词。
- 地图空状态区分“真的没有行程”和“已有安排但缺地点”。
- 时间线卡片只给下一步提示，不逐项堆叠所有缺失字段。

已处理：

## 第十四批同类问题：真实数据后的密度、焦点、移动端和 Agent 闭环

本轮复查基于真实高德 POI 和真实骑行路线数据，不再只看空状态。用户指出的五个点分别对应：窄屏、自动任务暴露、按钮样式、地图拥挤、拖动冗余。横向扩展后发现，同类问题主要集中在“有数据以后”的信息密度、编辑焦点、移动端抽屉、路线操作语义和 Agent 输出闭环。

### P1：地图有真实数据后仍承担过多任务，像复合面板而不是规划画布

1440px 复查时，地图卡同时包含范围切换、展开、最近加入反馈、搜索、真实高德地图、地点概览和路线概览。390px 复查时，地图下方地点卡被横向裁切，用户只能看到第二张地点卡的一部分。这和“地图布局拥挤”“地图要能拖动/缩放/搜索”属于同类问题：地图能力已经接入，但默认画布没有把“看地图”“选地点”“调整路线”分成清晰层级。

修复方向：
- 默认地图区只保留搜索、范围切换和地图本体；地点/路线概览改成可折叠的下方列表或当前选中摘要。
- 移动端地点和路线概览必须纵向单列，不允许被卡片宽度或网格截断。
- “已加入 Day N：地点”改为轻量 toast 或搜索结果内即时反馈，不长期占据地图卡层级。
- 展开地图进入明确的地图编辑模式，承担搜索、拖拽、缩放和地点选择；收起态侧重总览。

已处理：
- 地图底部概览在收起态移动端不再使用内部滚动，页面主滚动可以从地图概览区域继续进入时间线。
- 当前日期地点列表移动端改为单列，`sm` 以上再恢复多列，避免 390px 下地点卡被横向裁切。
- 新增前端测试固定 `map-overview-panel`、`map-day-place-list` 的移动端布局契约；浏览器证据见 `docs/design-audit-20260615/13-mobile-map-overview-after-workbench.png` 和 `14-mobile-map-scroll-after.png`。

### P1：路线卡仍暴露系统任务，且缺少行程可行性判断

当时复查的路线卡里仍显示“更新路线”，并把“路线、距离和耗时来自高德”作为一行说明展示给普通用户。用户关心的是交通方式是否合适、预计多久、是否会赶不上下一站，而不是是否触发了后台路线计算。当前有真实 `7.9 km / 32 分钟` 数据，但界面没有检查上一项结束时间 + 交通耗时是否晚于下一项开始时间。

修复方向：
- 路线按钮从“更新路线”改为“重新规划”或只在路线失败/用户改交通方式后出现。
- 高德来源信息降级为小型来源标识或详情内说明，默认摘要只显示方式、距离、耗时和风险。
- 增加时间冲突提醒：上一站结束时间 + 交通耗时超过下一站开始时间时，在路线卡和导出结果里提示“预计到达晚于下一项开始时间”。
- 人类手动编辑不产生 diff；Agent 改动的 diff 放在助手最后一条输出中。

已处理：
- 共享层新增 `detectTransportTimingConflict(from, to, leg)`，统一计算预计到达时间和晚到分钟数。
- 路线卡在交通耗时导致下一项赶不上时显示“预计 HH:mm 到达，晚于 HH:mm，需调整上一站停留或下一项开始时间”。
- 导出 Markdown 同步输出 `时间提醒：...`，保证最终交付行程也能看到风险。
- 手动路线调整只更新画布和导出，不生成右侧对话 diff。
- 第十五批继续把路线卡用户侧文案收口：`高德路线 / 估算路线 / 用户调整` 改为 `实时路线 / 参考路线 / 手动调整`，`更新路线 / 获取路线` 改为 `重新规划 / 规划路线`，并移除摘要层“路线、距离和耗时来自高德”说明。

### P1：移动端助手像临时覆盖层，不像完整的移动规划流程

390px 打开助手时，右侧面板覆盖主画布但仍露出左侧地图边缘；底部输入区和高德版权/地图控件在视觉上贴得很近。用户在移动端会认为这是临时抽屉，而不是稳定的“右侧对话栏”适配。这个问题和“考虑窄屏视频”同源：窄屏不能简单把桌面右栏压成覆盖层，还要定义移动端主画布与助手的切换关系。

修复方向：
- 移动端使用明确的全屏助手页或底部 sheet，打开后不露出主画布边缘。
- 助手顶部保留当前行程名、当前风格和返回画布动作，避免用户失去上下文。
- 输入区固定在底部时避开地图版权和系统安全区，保留足够触控间距。
- Agent 每轮完成后，最后一条 assistant 消息内展示本轮改动摘要和 diff，用户可回到画布确认。

已处理：
- 工作台三栏断点从 `2xl` 下调到 `min-[1360px]`。1399px 下助手以常驻右栏显示，画布宽度约 779px，不再出现灰色遮罩；1280px 仍保持两栏主画布优先。
- 390px 下助手抽屉改为全宽，并把抽屉/遮罩层级提升到高德控件之上，避免露出地图边缘或版权控件。
- 浏览器证据见 `docs/design-audit-20260615/16-desktop-assistant-rail-after.png`、`17-1280-assistant-drawer-threshold-after.png`、`20-mobile-assistant-over-map-after.png` 和 `21-desktop-assistant-rail-after-final.png`。

仍待处理：
- Agent 消息本身还需要结构化为“回复摘要 / 应用到画布的变更 / diff / 回到画布确认”，本轮只处理助手承载形态。

### P1：主画布内部滚动导致首屏、地图和时间线状态脱节

当前页面使用内部滚动容器。复查时浏览器窗口本身在顶部，但主画布已滚到活动/路线区域，导致截图一开始看不到地图首屏。用户在演示或窄屏录屏时容易失去“我现在看的是哪一天、哪一段”的上下文。这和顶部按钮/日期导航压缩问题属于同类布局问题：全局 header、地图、日志导航和编辑区域没有稳定的滚动锚点。

修复方向：
- 主画布滚动时保持日期导航或当前 Day 摘要 sticky，保证用户知道当前上下文。
- 地图和时间线之间建立明确的分区：地图是总览，时间线是编辑；不要让编辑表单把总览区域推到不可见后仍缺少上下文。
- 录屏宽度下优先保证“标题 + 当前日 + 地图缩略 + 时间线起点”在一次滚动内可理解。

### P2：活动编辑焦点仍然过重，和地图地点卡、时间线卡重复表达同一对象

当前同一个地点同时出现在地图下方地点卡、时间线卡、展开编辑表单中；用户点击地点卡或时间线卡后，很难判断主编辑入口到底在哪里。活动编辑表单虽然已从“大量输入框堆叠”收敛，但在真实 POI 场景下仍占据时间线顶部的大块空间，并把地图地址、电话、时间预算、内容、提醒全部展开。

修复方向：
- 时间线卡默认只呈现可扫读信息；编辑表单改为右侧/抽屉/内联单焦点，不和路线详情同时竞争。
- 地图地点卡只负责定位和选中，不承担编辑暗示；编辑动作回到时间线或专用编辑面板。
- POI 地址和电话折叠为“地点信息”二级内容，默认只显示名称、类型、时间、预算和关键提醒。

### P2：拖拽已能感知，但缺少拖动过程反馈和排序目标

卡片左侧已经使用拖动把手，且删除了“拖动”冗余文本；但当前视觉上只看到一个把手按钮和上移/下移按钮。用户会知道“可能可以拖”，但不知道拖动时会出现什么、是否能跨日期移动、路线是否会跟着更新。这和用户指出“应该能拖动景点”属于同类问题：需要完整排序交互，而不是只放一个把手。

修复方向：
- 添加拖动过程的占位线、目标位置高亮和拖后自动重算相邻路线。
- 明确跨日期移动是否支持；如果暂不支持，限制在当前日内并用日期 tab 提供移动入口。
- 上移/下移作为无障碍和键盘替代保留，但视觉权重低于直接拖拽。

### P2：助手 Skill 状态太薄，无法支撑“导入后参与对话和画布更新”的闭环

助手顶部目前只显示“当前风格 / 未选择风格 / 浏览风格”。这比之前干净，但还不足以表达用户导入的风格如何影响本轮 Agent 修改。Agent 消息区也只把“本轮改动”当普通文本行渲染，没有形成清晰的 diff 输出块。

修复方向：
- 当前风格状态栏显示已选风格名、移除入口和“查看影响”二级入口。
- 浏览风格使用独立 browse 面板，选择后回到对话，不长期挤占消息区。
- Agent 输出结构化为：回答摘要、应用到画布的变更、diff、可继续追问入口。
- 人类手动编辑只更新画布，不追加 diff；只有 Agent 一轮改动在对话末尾追加 diff。

已处理：
- Agent 回复中的 `本轮改动` 不再作为普通文本行渲染，而是进入独立可访问分组。
- 改动分组显示“已应用到画布”、参考风格和结构化清单，用户能区分普通回复和已写入画布的变更。
- Agent 结果消息携带 `changeSet`，前端可显示参考风格并保留本轮撤销快照。
- 用户可以在 diff 块点击“撤销本轮改动”，回退 Agent 本轮写入的画布状态；撤销后 diff 块显示“已撤销”。
- 前端测试覆盖运行 Agent 后出现 `role="group"`、`aria-label="本轮改动"` 的 diff 块、参考风格、清单项，并验证撤销会请求持久化恢复接口且新增活动从 Day 2 消失。

仍待处理：
- 单轮 Agent 撤销已通过 `POST /api/itineraries/:id/restore` 写回 SQLite；后续仍可扩展多版本 revision 历史和跨设备 undo 列表。
- 继续追问入口和回到画布确认定位还未补齐。
