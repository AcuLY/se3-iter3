## 1. 交付目标

迭代三需要交付：

- 完整可运行代码。
- 需求分析文档。
- 详细设计文档。
- 会议记录。
- Agent 优化文档。
- 优化过程中使用的评估数据集。
- 答辩用 PPT。

## 2. 产品主线

确定为：

**旅行风格 Skill 分享平台 + 多 Agent 行程规划工作台。**

业务闭环：

1. 用户规划行程。
2. 系统从行程、对话或外部游记提取旅行风格 Skill。
3. 用户编辑确认 Skill。
4. Skill 发布到广场。
5. 其他用户导入 Skill。
6. Agent 将旅行风格与新用户实际需求融合，生成新行程。
7. 新行程继续沉淀为新的 Skill。

## 3. 技术路线

- 前端：React + Vite + TypeScript + Tailwind CSS。
- 组件：shadcn 风格本地组件。
- 后端：Node.js + Express + TypeScript。
- 数据库：SQLite。
- 领域层：`packages/shared`。
- Agent 方向：OpenAI Agents SDK for TypeScript。
- 地图方向：高德 API，当前保留 mock fallback。

## 4. Agent 能力范围

必须支持：

- 多轮对话理解。
- 结构化行程生成与修改。
- Skill 融合。
- Skill 提取。
- 多 Agent 协作。
- 手动编辑协同。
- 高德方向工具调用。
- 外部 API 或脚本失败时降级。

明确不做：

- 真实支付、酒店/机票下单。
- 复杂多用户权限系统。
- 社交平台级审核流。
- 复杂商业推荐算法。

## 5. 前端形态

- 首页：较空的产品介绍首页。
- 左侧：功能入口和会话记录。
- 中间：行程编辑画布，顶部地图，下方日期导航和活动列表。
- 右侧：Agent 对话栏，支持 Skill 导入。
- 评估后台：仅用于答辩/开发，不作为普通用户主流程。

## 6. 当前暂缓的技术细节

### 下一批必须补齐

P0 核心流程必须补齐：
- 端到端旅行规划闭环：新建行程、手动编辑日期/活动/地点/路线/预算/同行人/备注、地图搜索填入、路线规划/修复、导出检查和导出结果定位都必须可跑通。
- Agent 闭环：DeepSeek 可用时走真实工具编排，不可用时确定性降级；用户能通过对话新增/替换地点、补路线、处理路线晚到、只读方案再选择执行；Agent 修改必须写入画布并在对话末尾显示 diff，手动编辑不生成 diff。
- Skill 闭环：支持导入标准 Skill、从当前行程/对话沉淀草稿、确认规则后发布、推荐/收藏/导入到当前行程，并持续影响右侧对话和中间画布；普通用户界面只展示旅行规划语义，不暴露 trace、工具名或主/子 Agent 内部名。
- 证据与交付闭环：API Key、本机 SQLite、本地对象存储、评估数据集、Agent 优化文档、需求/详细设计/会议记录和答辩 PPT 必须能支撑最终验收。

P1 展示质量补齐：
- 地图与长行程体验继续抛光：失败原因自动诊断、路径段与跨天总览的视觉完成度继续加强。
- Skill 与偏好只补对核心规划有帮助的反馈：风格影响范围、规则取舍、版本记录、偏好来源和最近影响证据；不做规则评分、社区资产治理和完整资产回溯。
- 评估后台继续保留主/子 Agent 编排证据，但普通工作台不回流技术解释卡片。

P2 暂缓：
- 社区资产治理、规则质量评分、跨行程全局偏好、更细粒度流式过程和完整资产回溯。
- 最终交付前仍需引入 product-design 的严格 audit/design-qa 流程，对 P0/P1 完成度不足项返工。
