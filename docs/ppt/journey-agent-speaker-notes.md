# Journey Agent 答辩逐页讲稿

预计总时长：7 分 30 秒  
语速假设：较快语速，约 250 到 270 个中文字符/分钟  
使用方式：PPT 页面保持少字，讲稿用于口播；讲完第 14 页后自然切到浏览器做实机演示

## 01 / 14 封面与产品定位（0:25）

今天带来的是 Journey Agent，一个给自由行规划用的 Agent 工作台。用户可以像聊天一样说需求，最后拿到一份能继续改的行程；聊天只是入口。右侧截图是一次规划完成后的界面：地图、路线、行程画布和 Agent 的执行结果都在同一个地方。

## 02 / 14 产品背景（0:30）

这一页先从旅行规划本身出发。真实规划通常会先确定目的地、日期和同行人，再加入预算、节奏、禁忌等个人约束；随后还要在景点、餐饮、住宿之间做取舍，并根据天气、距离和营业时间反复校准。实际旅行结束后，用户还会形成对这类旅行风格的判断，例如是否喜欢慢节奏、少换乘或亲子友好路线。一次性的问答结果不够用，用户最后需要一份能随着现实约束和真实体验继续积累的行程。

## 03 / 14 产品总览：地图画布（0:25）

这一页先看地图画布。地点分布、路线距离和当日范围都直接可见，用户不用在聊天文本、地图软件和表格之间来回对照。这里想强调的是，行程会进入可以继续编辑的产品界面，不会停留在一段建议里。

## 04 / 14 产品总览：行程编排（0:25）

这一页看具体行程编排。Day 标签、活动卡片、时间段和交通衔接都集中在同一个画布里，用户后续可以继续替换地点、调整顺序和修改节奏。系统最终维护的是一份结构化行程，聊天文本只是入口。

## 05 / 14 产品总览：Agent 协作（0:30）

这一页看 Agent 执行结果的两个证据。左侧是工具调用记录，可以看到它先搜索地点，再查询天气，最后规划交通路线；右侧同一张截图里同时保留 Agent 给用户的最终输出和本轮改动 diff。这里要强调的是：Agent 不是只给一段建议，而是把工具过程、用户可读结论和写入结果都留在界面里。

## 06 / 14 工作流程（0:35）

整体流程分四步。用户先用自然语言说旅行意图；Agent 调用地点、路线、天气和行程编辑工具，把这些意图转成结构化改动；用户在画布里确认、继续调整或导出；最后，稳定偏好写入记忆，有用的方法保存成 Skill。对话只是入口，真正留下来的是行程、记忆和 Skill。

## 07 / 14 执行链（0:40）

第一部分是执行链。Agent 运行时不会只让模型写一大段建议。它会先读取当前行程、已挂载的 Skill 和 saved memories，再用工具调用表达具体动作。工具结果先写进行程对象，随后生成 diff，让用户看到这一轮到底改了什么。运行过程也会留下 trace，之后恢复会话时还能查到当时调用了哪些工具。

## 08 / 14 上下文管理与压缩（0:40）

第二部分是上下文管理。旅行规划聊久了，历史会越来越长，如果每次都把全部内容塞给模型，速度会慢，噪声也多。我们的做法不是让模型读完所有历史，而是先做过滤：最近几轮保留原文，更早内容由 ConversationContextService 压缩成摘要，只留下已定安排、约束和待办。这里的多 Agent 不是多个独立智能体并行决策，而是角色化分工：ContextAgent 处理历史，PlannerAgent 负责行程改动，TransportAgent 负责交通相关步骤。用户侧看到的是和本轮任务有关的信息，无关内部上下文被屏蔽掉。

## 09 / 14 多轮连续性与长期记忆（0:45）

这一页区分两层连续性。会话连续性面向当前行程，依赖会话历史、行程关联和压缩摘要，定位上一轮安排对应的日期、活动或改动。长期偏好面向后续规划，例如少换乘、亲子友好、雨天优先室内，通过 MemoryService 和 `/api/memories` 保存，并在偏好设置中集中管理。

## 10 / 14 用户给 Agent 添加能力（0:30）

第四部分是 Skill。它解决的不是单次规划，而是旅行经验复用。用户可以从 Skill Plaza 复刻他人的成熟路线方法，也可以用 Skill Creator 把自己的偏好、限制和示例沉淀成规则。之后这些 Skill 会进入行程上下文，Agent 新增、替换或复盘安排时都能读取，不需要每次从空白 Prompt 重新描述。

## 11 / 14 Skill Plaza（0:30）

Skill Plaza 是第一种入口。用户可以在广场里浏览已有旅行风格，把适合当前行程的 Skill 导入并挂载。挂载后，这些风格会进入当前行程上下文。后面 Agent 做规划、修改或解释时，都会参考它。所以 Skill 会参与运行，作用超过一张静态模板。

## 12 / 14 Skill Creator（0:35）

Skill Creator 是第二种入口，用来保存用户自己的经验。它通过引导式对话收集适用场景、规则、限制和示例，再把这些信息整理成新的 Skill。中间如果缺字段，系统会继续追问补齐，用户不用直接面对复杂配置。创建完成后，用户可以发布、收藏，也可以把新 Skill 挂到行程上。

## 13 / 14 从坏案例优化 Agent（0:45）

最后讲一下优化方式。我们把坏案例变成边界，避免只靠堆功能解决问题。比如用户只想查询时，Agent 不应该改行程，所以拆出了只读工具。取消交通段时不能误删活动，所以交通删除和活动删除分开。路线晚到时，系统不能只提醒一句，还要能延后、缩短或顺延安排。每一条约束都来自一个具体问题。

## 14 / 14 产品价值总结（0:25）

总结一下，Journey Agent 把重点放在行程生产上，聊天答案只是入口。结果能改，过程有记录，偏好和 Skill 能带到下一次。接下来我会切到浏览器做实机演示，跑一遍从对话修改行程、查看执行记录，到再次使用偏好和 Skill 的流程。
