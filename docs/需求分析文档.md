# 需求分析文档

## 1. 项目背景

本项目面向自由行规划场景，目标是把大模型对话能力落到可编辑、可追踪、可复用的旅行规划产品中。系统不只生成一段文本行程，而是把 Agent 的规划结果写入结构化行程画布，用户可以继续手动调整日期、活动、地点、路线、预算和偏好。

迭代三的产品主线是：

**旅行规划工作台 + 旅行风格 Skill 广场 + 可跨对话沉淀的 Agent 记忆。**

系统需要解决三个核心问题：

- 对话不是一次性问答，用户需要恢复历史行程会话、清除不需要的会话记忆，并查看 Agent 本轮对画布造成的改动。
- 旅行偏好、历史对话和已导入 Skill 需要在多轮、多次对话中复用，避免用户反复说明同样的风格约束。
- 用户需要能把行程、对话或外部游记沉淀成 Skill，并把 Skill 添加给 Agent，影响后续规划。

## 2. 用户角色

- 旅行规划用户：创建行程，手动编辑活动，使用地图和路线能力，和 Agent 对话补全或调整计划。
- 旅行风格创作者：从当前行程、最近对话或外部攻略中整理旅行风格，确认后发布到 Skill 广场。
- 开发与验收人员：根据 Agent trace、评估数据集和文档复盘优化过程。

系统面向普通用户的主界面应使用旅行规划语言，不暴露主 Agent、子 Agent、tool call、trace 参数等内部实现词。

## 3. 核心业务闭环

1. 用户创建行程，填写出发点、日期、同行人、预算、备注和偏好。
2. 用户在工作台中手动添加日期、活动、地点和路线，也可以通过地图搜索填入 POI。
3. 用户从 Skill 广场选择旅行风格，或通过 Skill Creator 创建自己的风格。
4. 用户向 Agent 发送自然语言请求，例如新增地点、替换活动、补全路线、处理晚到冲突或只请求调整方案。
5. Agent 读取当前行程、已导入 Skill、用户偏好和会话上下文，调用地点、路线、天气、时间调整等能力，生成结构化改动。
6. 系统把 Agent 改动写入画布，并在右侧助手回复末尾展示“本轮改动”；用户手动编辑不生成 Agent diff。
7. 用户可以把当前行程和对话沉淀为新的旅行风格 Skill，再发布、收藏或导入到其他行程。

## 4. 功能需求

### 4.1 行程工作台

- 支持创建、浏览、归档、恢复、删除行程。
- 支持编辑行程基础信息：出发点、日期范围、同行人、预算、偏好和备注。
- 支持按天管理活动：添加日期、添加活动、编辑活动、删除活动、移动活动、跨天移动活动、调整顺序。
- 支持地点字段和活动类型维护，包括景点、餐饮、住宿、交通、自由活动等。
- 支持导出 Markdown 行程，并在导出前检查缺地点、缺完整时间和待计算交通。

### 4.2 地图、地点和路线

- 支持地点搜索，并把 POI 名称、地址、行政区、电话、营业时间、人均消费和坐标写入活动。
- 支持按当前日期或全部行程查看地点和路线。
- 支持规划步行、公交/地铁、驾车、骑行等交通方式。
- 支持用户手动覆盖路线距离、耗时、费用、说明和备注。
- 支持路线时间风险提示，例如上一项结束时间加交通耗时会晚于下一项开始时间。
- 支持路线失败后的修复入口，帮助用户编辑端点、重新规划或手动记录路线。

### 4.3 Agent 对话与会话管理

- 右侧助手支持流式运行、停止本轮处理、显示失败原因和保持画布不变。
- Agent 可以新增活动、替换地点、补全路线、比较交通方式、取消指定交通段、移动活动、删除活动、更新行程信息和修复路线晚到。
- Agent 每轮只把实际写入画布的变化展示为“本轮改动”；导入 Skill、手动编辑和只读方案不混入 diff。
- 用户可以恢复某个行程的历史对话，搜索历史 conversation，并继续在同一行程上下文中规划。
- 用户可以按行程清除 Agent session 和 trace，避免旧会话继续影响后续规划。

### 4.4 用户如何管理对话与工具能力

当前产品不要求普通用户维护底层工具注册表，而是通过旅行规划入口管理 Agent 可以使用的能力：

- 对话管理：系统保存 Agent session、trace 和行程关联关系；用户可通过历史入口恢复会话，通过清除记忆删除当前行程的 session 和 trace。
- 工具能力管理：地点搜索、路线规划、天气查询、路线修复、Skill 导入、Skill Creator 和偏好设置都以产品功能形式出现。用户选择地点、导入风格、编辑偏好或要求路线修复时，等价于给 Agent 提供可用工具和上下文。
- 执行反馈：底层 tool call 不直接暴露在普通工作台中，用户只看到“已添加地点”“已补全交通路线”“已顺延活动”等可理解结果。

### 4.5 跨对话知识共享

- 系统提供 saved memories，用于保存用户长期偏好和可复用约束。
- 系统保存与行程关联的历史对话、会话摘要和偏好摘要，后续 Agent 运行会把这些上下文注入提示输入。
- 用户可以在偏好设置中查看、编辑和清除偏好内容。
- Agent 不把内部摘要直接塞进聊天流，而是把它用于规划判断；普通用户看到的是行程和偏好是否被应用。

### 4.6 用户向 Agent 添加 Skill

- 用户可以在 Skill 广场浏览、收藏、推荐和使用旅行风格。
- 用户可以把 Skill 挂载到当前行程，Agent 后续规划会读取已挂载 Skill 的规则、禁忌和适用场景。
- 用户可以粘贴旅行风格内容导入 Skill，也可以从当前行程、最近对话或外部攻略中提取 Skill 草稿。
- Skill Creator 以引导式对话补齐名称、描述、规则、禁忌和示例，用户确认后发布。
- Skill 发布后可继续更新，并保留版本记录；收藏和导入次数不改变内容版本。

## 5. 非功能需求

- 可维护性：前端、后端和共享领域逻辑分层，核心类型集中在 `packages/shared`。
- 可追踪性：Agent session 和 trace 保留运行证据，支持问题复盘。
- 可控性：用户停止流式运行后，服务端不应继续落库或隐藏修改画布。
- 一致性：Agent 修改、手动编辑、导出检查和路线风险提示使用同一套结构化行程数据。
- 安全边界：当前不做真实支付、票务下单、复杂权限系统和社区审核流。
- 外部服务边界：地图和路线能力依赖高德 Web Service Key；无 Key 或外部接口异常时应在界面上以可理解错误提示收束，不把失败伪装成真实路线。

## 6. 验收标准

- 用户可以完成“创建行程 -> 手动添加活动 -> 搜索地点 -> 规划路线 -> 导出检查”的闭环。
- 用户可以完成“导入/创建 Skill -> 添加到当前行程 -> Agent 按风格改写规划 -> 查看本轮改动”的闭环。
- 用户可以完成“恢复历史会话 -> 查看或清除偏好记忆 -> 再次运行 Agent”的闭环。
- Agent 对纯路线、纯详情、活动更新、地点替换、交通取消、活动移动和活动删除等请求不应额外新增无关活动。
