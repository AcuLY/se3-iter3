## 总体判断

用户标出的 5 个问题不是孤立 UI 小问题，而是同一组设计债的表现：

1. 主画布、地图、助手、编辑表单在同一屏竞争主任务。
2. 手动编辑能力已经存在，但编辑对象、编辑区域和结果反馈的边界不稳定。
3. 地图具备真实能力，但“概览 / 编辑 / 路线检查”的任务模式还不够清晰。
4. 普通用户界面仍残留部分实现口径、格式口径或后台口径。
5. 响应式断点还在按布局能否放下三栏判断，而不是按用户任务是否仍清晰判断。

## P1 问题

### 1. 1399px 三栏布局让主画布过窄

证据：`App.tsx` 的 `shellGridClass` 在 `min-[1360px]` 就启用 `260px + 1fr + 360px` 三栏；截图中主画布约 779px，地图、Day 栏和活动卡都被压缩。  

影响：
- 右侧助手常驻方向是对的，但 1360-1439px 这段宽度下，主画布已经像窄屏编辑器。
- 地图和 Day 栏不得不使用更多换行，活动编辑态更容易把内容推到首屏外。
- 后续加入更多路线检查、冲突修复、景点信息后会继续拥挤。

建议：
- 三栏常驻断点提高到 `1536px` 或按容器宽度判断中间画布最小宽度，例如主画布低于 `880px` 时助手改为抽屉。
- 1360-1535px 保持两栏：左侧导航 + 主画布；右侧助手通过顶部按钮打开。
- 如果必须在 1399px 常驻助手，右栏应收缩为状态栏/对话抽屉，而不是完整 360px rail。

### 2. 地图模式还没有形成稳定任务边界

证据：默认地图上同时出现路线、地点标签、缩放控件、当前日期/全部行程切换、编辑地图按钮和底部概览。用户已反馈地图看起来像只读，也不确定能否展开、拖动、缩放或搜索地点。  

影响：
- 地图既像主画布，又像路线预览，但用户不知道当前可直接操作什么。
- 搜索地点只在“编辑地图”后出现，默认态缺少“为什么现在不能搜”的清晰模式边界。
- 长行程全程概览里 Day 卡可点击，但具体地点还只是静态列表，不能直接定位到活动编辑。

建议：
- 明确三种模式：路线概览、地图编辑、路线检查。
- 路线概览只保留路线轮廓、当天地点摘要和“编辑地图”入口。
- 地图编辑提供搜索、候选点、已选点、路线段和地图画布，且至少一个管理区域始终可见。
- 全部行程概览的地点/路线段要可点击，点击后切换到对应 Day 并选中活动或路线。

### 3. 活动/路线详情内联展开导致编辑上下文不稳定

证据：打开活动编辑或路线详情后，页面滚动位移明显，顶部 header 和部分侧栏元素出现负坐标；编辑表单、地图、Day 栏、助手输入同时在一个视口里竞争。  

影响：
- 用户点击“编辑”后不知道自己是否进入了独立编辑状态，还是只是展开了一块内容。
- 编辑区从页面中段开始，首屏只能看到表单上半部，保存/应用动作不一定可见。
- 右侧助手仍处于同等输入优先级，纯手动规划时干扰主任务。

建议：
- 中间画布建立稳定 inspector：选中活动/路线后，详情固定在 Day 栏下方或作为画布内右侧 inspector，而不是普通流式卡片随页面滚动。
- 保持同一时间只有一个详情对象，且进入编辑后自动定位到详情顶部。
- 右侧助手在手动编辑详情时降级为空闲协作态，只保留输入和最近 diff，不抢占编辑说明空间。

### 4. 路线详情的“数据来源”和“我的调整”仍然混在同一层

证据：路线详情展开后包含交通方式、重新规划、距离、耗时、费用、路线步骤、怎么走、备注、应用调整。截图中右侧“应用调整”区域被拉伸成过大的竖向块，说明布局在当前宽度下不稳定。  

影响：
- 用户不清楚哪些是高德/系统给出的实时数据，哪些是自己覆盖的手动值。
- “重新规划”和“应用调整”同时存在，容易误解为两个保存动作。
- 过大的应用按钮破坏路线卡密度，长行程下会严重拉长页面。

建议：
- 路线详情拆为两层：只读路线结果和手动覆盖表单。
- “重新规划”只属于路线结果区域；“应用调整”只属于手动覆盖区域。
- 应用按钮放在表单 footer，不使用 `xl:self-stretch` 这类会被父容器高度放大的布局。

### 5. 深链接和页面状态不完全一致

证据：直接访问 `#/plaza` 抓到的是首页；通过侧栏进入后 URL 为 `#/skills` 才显示 Skill 广场。  

影响：
- 录屏、刷新、答辩跳转时容易出现地址和页面不一致。
- 用户分享或刷新某个功能页时可能回到错误页面。
- 对“最终可上线”的产品完整性有明显影响。

建议：
- 明确页面路由表，统一 `Skill 广场` 的 hash path，避免 `/plaza`、`/skills`、内部 state 混用。
- 每个主功能入口都应支持直接访问、刷新恢复和浏览器前进后退。

## P2 问题

### 6. 非工作台页面仍显示完整会话历史

证据：Skill 广场、创作 Skill、偏好设置都保留左侧完整会话列表和归档/删除按钮。  

影响：
- 在风格广场和偏好设置这类页面，历史列表成为干扰信息。
- 归档/删除行程属于工作台历史管理，不一定应该在每个功能页常驻。

建议：
- 左侧保留主功能导航，但历史区在非工作台页面折叠为“最近行程”简版，或只在当前行程页完整展开。
- 归档/删除操作移入行程历史页或 hover/更多菜单，避免普通浏览页持续暴露维护动作。

### 7. Skill 广场首屏信息密度偏低

证据：1399px 下推荐卡从约 `y=169` 开始，但页面右侧顶部仍有较大空白；卡片主信息只有两张，筛选和“推荐风格”badge 占据较强层级。  

影响：
- 广场像展示页，不像可快速发现、比较、导入的工具页。
- 旅行风格的适用人群、规则摘要、与当前行程匹配原因没有被优先展示。

建议：
- 广场首屏使用更密集的推荐瀑布流或两列卡片，但每张卡优先展示“适合什么行程”和“为什么推荐给当前行程”。
- `推荐风格` badge 降级为筛选状态，不和主动作竞争。
- 当前已导入的风格要在页面顶部有状态栏，避免用户只在右侧助手里找。

### 8. 创作 Skill 页面把高级格式暴露得过早

证据：创作页同时显示来源文本、确认草稿、Markdown 预览、发布检查。`# Extracted Travel Style` 等格式化内容在普通创作者流程中过早出现。  

影响：
- 普通用户的任务是“把游记/行程沉淀为旅行风格”，不是检查 Markdown。
- 发布检查是必要能力，但现在像开发校验面板。

建议：
- 改成分步流程：来源 -> 提取结果 -> 规则确认 -> 发布。
- Markdown 预览放入“高级 / Skill.md”折叠区，只在导入、调试或答辩时展开。
- 发布检查只显示阻塞项和一键修复建议，不长期展示完整校验清单。

### 9. 偏好设置页面仍像后台管理页

证据：页面显示行程偏好输入框、保存按钮、会话记忆空状态，但没有偏好来源、最近应用场景、清除影响范围等解释。  

影响：
- 用户知道这里能管理偏好，但不知道这些偏好会影响哪里。
- “会话记忆”已经从助手区移出是正确的，但设置页还没有把它产品化。

建议：
- 偏好拆成结构化芯片：节奏、预算、餐饮、住宿、禁忌、同行人。
- 会话记忆显示来源时间、最近影响的行程和清除范围。
- 清除记忆要说明只影响当前行程还是全部行程。

### 10. 图标按钮触控目标和含义仍不稳定

证据：历史项归档/删除按钮宽约 28px；当前风格 chip 的移除按钮约 20px；移动端“添加活动”显示为“添加”。  

影响：
- 触控目标低于常见 44px 建议，移动端和触屏设备误触风险高。
- “添加”在 Day 上下文里还算可猜，但长远会与添加日期、添加地点、添加风格混淆。

建议：
- 危险或维护动作放入 36-40px 图标按钮或更多菜单；触屏断点保证 44px 点击区域。
- 移动端主动作保留对象名，至少使用 `加活动`、`加日期`、`加地点`，不要只写“添加”。

### 11. 地图第三方控件和业务摘要缺少安全边界

证据：移动端截图中高德版权、比例尺、缩放控件和地点标签贴近业务内容；弹窗曾需要提高到 `z-[1200]` 才避免穿透。  

影响：
- 地图控件容易压住业务说明或被业务浮层压住。
- 后续地图搜索、路线检查、全屏编辑会继续遇到 overlay 层级问题。

建议：
- 建立统一 z-index token：地图 SDK、地图内业务浮层、sticky 上下文、助手抽屉、全局 modal 分层。
- 地图容器为版权和缩放控件预留 padding，不把业务摘要贴到底部。

## 建议修复顺序

1. 先修路由和断点：保证主功能页可刷新恢复，1360-1535px 不再牺牲主画布。
2. 再重构地图模式：概览更轻，编辑模式可管理地点/路线，全部行程能定位到具体活动。
3. 接着重做活动/路线 inspector：选中对象详情稳定定位，不再靠流式卡片撑开页面。
4. 然后收口路线详情：只读结果和手动覆盖分层，修掉应用按钮拉伸问题。
5. 最后清理二级页面：Skill 广场提密度，创作页分步化，偏好设置结构化。

证据：`厦门海边松弛风格 8695` 使用了偏峡谷/公路感图片，与厦门海边语义不匹配；标题带随机数字后缀；卡片没有解释为什么适合当前杭州行程。

影响：
- Pinterest 式卡片依赖图片可信度，错误图片会比无图更伤害产品质感。
- 随机后缀像测试数据或内部 ID。
- 用户只能看到收藏数和标签，不能判断是否值得导入到当前行程。

建议：
- Skill 卡图片需要与目的地或风格标签匹配，没有可靠图片时使用稳定的本地占位视觉，不用随机远程图。
- 去掉标题随机后缀，冲突名用 `草稿`、`副本`、`版本时间` 等用户可理解语义处理。
- 卡片增加 `适合当前行程的原因`，例如“当前行程偏 citywalk，且已有咖啡/湖边路线”。

处理记录：
- Skill 卡片不再按数组下标轮换远程图片，改为根据风格内容生成稳定的本地语义视觉：街区漫步、亲子博物馆、海边松弛、雨天室内等。
- 卡片新增 `适合当前行程` 区块，展示匹配当前偏好、已安排内容、目的地或首条规划规则，用户可以判断是否值得用于当前行程。
- 广场顶部新增当前行程状态栏，显示当前行程已使用的旅行风格；导入或移出风格后，广场和助手入口使用同一套展示名。
- 展示层统一清理尾部 3 位及以上随机编号，例如真实旧数据 `厦门海边松弛风格 8695` 会显示为 `厦门海边松弛风格`，底层保存名不被破坏。
- 收藏列表继续可用，并在顶部显示个人收藏数量；用户导入或创作的风格即使发布后也可编辑标签，官方广场风格仍不暴露标签编辑。
- 前端测试覆盖推荐卡语义视觉、当前行程适配理由、收藏列表、官方卡不显示标签编辑、自定义导入风格显示标签编辑。
- 浏览器复查截图：`screenshots/20260615-skill-plaza-credible-after.png`；控制台 `error/warn` 为 0。

### 新发现 5：创作 Skill 仍过早暴露 Markdown 和校验面板

证据：创作页首屏下方直接出现 `# Extracted Travel Style`、`发布检查`、格式头、name、description 等内容；当前页面可见 2 个输入框、5 个文本域。

影响：
- 普通用户的任务是沉淀旅行风格，不是理解 Skill.md 结构。
- 校验清单长期可见，会把创作流程拉回开发/答辩工具感。
- 页面纵向高度达到约 1678px，确认草稿阶段显得像表单堆叠。

建议：
- 先做四步流程：`选择来源 -> 提取风格 -> 确认规则 -> 发布`。
- 默认只展示风格名称、适用场景、规划规则和禁忌；`Skill.md` 原文放入高级折叠区。
- 发布检查只在存在阻塞时展示阻塞项；全部通过时只显示一个轻量状态。

处理记录：
- 已将创作页改为 `来源 -> 提取 -> 确认规则 -> 发布` 的分步流程，默认草稿态不再显示 `Skill 正文`、`发布检查` 或 `--- name:` 格式头。
- 草稿确认阶段只保留普通创作者需要判断的字段：名称、标签、说明、规划规则和避免安排；`Skill.md` 正文与原文预览移入 `编辑高级内容` 折叠区。

- 前端测试覆盖“提取后默认隐藏 Skill 正文/发布检查/Skill.md 原文”和“清空规划规则后显示发布检查并禁用发布”。

### 新发现 6：偏好设置还不是用户可管理的偏好系统

证据：偏好设置仍是一个 textarea + 一个重复摘要条；会话记忆为空时只有 `暂无会话记忆`。

影响：
- 用户不知道偏好会影响风格推荐、助手回复、路线规划中的哪一部分。
- 单个 textarea 不利于后续做结构化 Agent 上下文，也不利于用户精确删除某类偏好。
- 会话记忆移出助手是正确方向，但设置页还没有说明来源、范围和影响。

建议：
- 偏好拆成结构化组：节奏、预算、餐饮、住宿、交通、同行人、禁忌。
- 会话记忆展示来源时间、来源对话、影响范围，并提供按当前行程清除。
- 设置页不再重复展示 textarea 内容，而是展示结构化偏好 chips 和最近应用记录。

处理记录：
- 偏好设置页新增结构化偏好组：节奏与强度、地点兴趣、餐饮与停留、交通与体力、约束与禁忌、其他偏好。
- 用户可以通过 `添加偏好` 输入框新增偏好，也可以在每个 chip 上精确移除；原 `行程偏好` textarea 保留为批量编辑入口，不再是唯一管理方式。
- 会话记忆卡片显示当前行程范围、最近更新时间、来源、偏好摘要、最近对话和影响范围，明确这些记忆会影响助手回复、风格融合、地点取舍和路线调整。
- 空记忆状态不再只是 `暂无会话记忆`，补充说明助手会先参考上方行程偏好，产生对话后再显示来源和最近影响。
- 前端测试覆盖结构化分组、添加偏好、移除偏好、当前行程范围、来源和影响范围；浏览器复查截图：`screenshots/20260615-preference-settings-structured-after.png`，控制台 `error/warn` 为 0。

### 新发现 7：助手空白态仍像功能说明区

证据：右侧助手空白态显示一段静态说明：`发送后会更新画布；本轮改动会在下一条消息里列出，可定位和撤销。`

- 右侧当前风格 chip 的移除按钮约 20px，触控目标偏小。

- 右侧 `浏览风格` 面板接入 Skill 广场同一套 `适合当前行程` 推荐理由，移除旧的机制说明文案。

### 新发现 8：保存时间、历史时间和数据来源语义仍需统一

证据：工作台顶部显示 `已保存 09:18`，左侧当前行程显示 `最近 06/15 07:29`。

影响：
- 用户会误以为当前修改没有同步到历史记录。
- 如果一个是保存时间、一个是最近打开时间，需要明确命名；如果都是更新时间，则数据同步有问题。

建议：
- 统一为 `已保存 HH:mm` 和 `最近打开/最近编辑` 两种明确语义。
- 历史列表使用后端 `updatedAt` 时，需要在保存后同步更新当前行程条目。
- 为“保存后侧栏时间更新”补充测试。

处理记录：
- 保存提交统一经过 `commitSavedItinerary`，使用行程对象的 `updatedAt` 作为顶部 `已保存 HH:mm` 的时间来源，并同步更新左侧行程列表。
- 手动添加/编辑/删除/排序/移动活动、路线调整、天气/路线后台写回、行程信息修改、导入/移除旅行风格、Agent 写画布和 Agent 撤销都接入同一保存提交入口。
- 左侧历史时间文案从 `最近 MM/DD HH:mm` 改为 `最近编辑 MM/DD HH:mm`，无效时间显示为 `最近编辑时间未知`，不再混淆保存状态和历史编辑时间。
- 异步天气/路线写回继续基于最新画布合并，避免后台返回覆盖用户刚刚手动编辑的内容。
- 前端测试覆盖“手动添加活动后顶部显示 `已保存 HH:mm`、侧栏显示 `最近编辑 MM/DD HH:mm`，且不再出现旧的 `最近 MM/DD...`”；浏览器复查 `1181x884` 工作台，添加活动后控制台 `error/warn` 为 0，页面无旧时间文案和无内部泄漏词。
