## 1. 交付目标

迭代三需要交付：

- 完整可运行代码。
- 需求分析文档。
- 详细设计文档。
- 会议记录。
- Agent 优化文档。
- 优化过程中使用的评估数据集。
- 答辩用 PPT。

## 2. 产品主线

确定为：

**旅行风格 Skill 分享平台 + 多 Agent 行程规划工作台。**

业务闭环：

1. 用户规划行程。
2. 系统从行程、对话或外部游记提取旅行风格 Skill。
3. 用户编辑确认 Skill。
4. Skill 发布到广场。
5. 其他用户导入 Skill。
6. Agent 将旅行风格与新用户实际需求融合，生成新行程。
7. 新行程继续沉淀为新的 Skill。

## 3. 技术路线

- 前端：React + Vite + TypeScript + Tailwind CSS。
- 组件：shadcn 风格本地组件。
- 后端：Node.js + Express + TypeScript。
- 数据库：SQLite。
- 领域层：`packages/shared`。
- Agent 方向：OpenAI Agents SDK for TypeScript。
- 地图方向：高德 API，当前保留 mock fallback。

## 4. Agent 能力范围

必须支持：

- 多轮对话理解。
- 结构化行程生成与修改。
- Skill 融合。
- Skill 提取。
- 多 Agent 协作。
- 手动编辑协同。
- 高德方向工具调用。
- 外部 API 或脚本失败时降级。

明确不做：

- 真实支付、酒店/机票下单。
- 复杂多用户权限系统。
- 社交平台级审核流。
- 复杂商业推荐算法。

## 5. 前端形态

- 首页：较空的产品介绍首页。
- 左侧：功能入口和会话记录。
- 中间：行程编辑画布，顶部地图，下方日期导航和活动列表。
- 右侧：Agent 对话栏，支持 Skill 导入。
- 评估后台：仅用于答辩/开发，不作为普通用户主流程。

## 6. 当前暂缓的技术细节

## 8. 产品完成度缺口追踪

### 2026-06-14 已推进

- 中间行程画布的活动行改为“摘要优先、编辑展开”的结构，避免大量输入框默认堆叠。
- 活动支持手动上移、下移、拖拽排序、跨天移动，用户不依赖 Agent 也能调整每天路线。
- 地图支持当前日期/全部行程切换，全部行程按 Day 分组展示，点击 Day 路线卡可回到对应日期。
- 当前日期地图中的地点卡可反向选中活动，选中后活动卡高亮并展开编辑字段；高德 Marker 点击也接入同一选中行为。
- 每段交通支持展开细节，用户可覆盖距离、耗时、费用、怎么走和备注；覆盖结果会持久化并进入导出内容。
- 每段交通开始展示路线来源状态：高德路线、估算路线或用户调整；导出文档同步标明 fallback 来源，避免用户误以为参考值一定是实时路线。
- 内部实现文案已从普通用户工作台移除，例如 Agent 输入、Agent 协作、高德地图已加载、交通备注等。
- Agent 一轮修改的 diff 已移到右侧对话的本轮助手回复末尾；用户在画布里的手动编辑只更新行程，不生成对话 diff。
- 工作台完成第一批同类 UI 问题修复：1280/1024 录屏宽度下不再强行三栏挤压，Agent 改为超宽常驻、窄屏顶部按钮打开抽屉；行程信息默认折叠为摘要；天气刷新改为自动触发，不再向普通用户暴露“更新天气”；地点搜索集中到地图区；地图控件与地点/路线列表从地图画面中拆出，避免遮挡拖动缩放；活动排序把手改为纯图标，去掉冗余“拖动”文本。
- 工作台第二批同类问题分析与修复：把问题归类为内部状态暴露、按钮层级混杂、地图信息过密、表单字段缺少任务语义、运行状态长期残留。对应调整为：顶部保存状态和操作按钮分离且样式一致；行程信息编辑从横向密集字段改为分组表单；地图摘要从长句改成状态 chip 并放大默认地图画布；路线细节改为“路线数据 / 我的调整”；Agent 进度只在运行中展示，完成后的 diff 仍只出现在本轮助手回复末尾。
- 工作台第三批同类问题分析与修复：新增 `docs/design-audit-20260614/design-audit.md`，用截图证据归纳“助手遮挡主画布、开发/管理能力混入普通入口、编辑型输入框默认暴露、空状态提示重复、Skill 浏览信息过密”等问题。对应调整为：1280px 以上恢复固定右侧助手栏，1280px 以下才使用抽屉；评估后台移到左侧底部“开发与答辩”；助手内 Skill 浏览改为轻量“使用风格”卡片，不直接展示规则和禁用项；Skill 广场默认只做浏览、收藏和用于当前行程，Skill.md 导入和标签编辑改为按需展开；地图空状态删除重复底部说明。
- 地点搜索开始携带更完整的旅行规划信息：高德/本地 POI 结果会保留行政区、分类、电话、营业时间、人均消费和照片字段；用户从地图搜索加入活动或在活动详情里选择地点后，这些信息会进入可编辑活动，并同步进入导出 Markdown。Agent 的地点补全工具也复用同一映射，避免生成只有名称/地址的空壳地点。
- 工作台第四批同类问题分析：当前复杂编辑态的问题不再只是单个文案或按钮，而是断点、焦点和重复控制。1280px 固定三栏会挤压主画布；地图地点卡、日程卡、活动表单和路线卡同时表达同一对象；活动编辑与路线细节可同时展开；默认“新的活动 / 杭州”会让路线标题和地图卡不可读；批量路线和单段路线计算入口层级接近；Skill 浏览仍挤在对话流里。后续 UI 收敛优先级定为：先改断点和画布宽度，再改编辑焦点，再改新增活动流程，再改路线入口，最后统一保存语义、选中态颜色和字段单位。
- 交通规划补齐路线步骤链路：高德路线和本地估算路线都会返回 `routeStatus`、`fallbackReason` 和结构化 `steps`；API、Agent 工具调用、前端本地 fallback 保存同一组字段；路线详情面板展示“路线步骤”；导出 Markdown 会输出分段路线说明，避免只看到总距离和总耗时。
- 工作台第四批 UI 收敛已完成第一轮实现：1280-1399px 不再强制三栏，右侧助手到 `2xl` 才常驻；活动详情和路线详情保持单一编辑焦点；助手内 Skill 浏览改为临时选择浮层，选择风格后回到对话，移出风格时保留面板便于继续管理。
- 工作台第五批同类问题分析与修复：把“无意义文本”和“地图拥挤”的根因继续细化为默认值冒充真实数据、路线端点判定缺失、显示名缺少降级策略。对应调整为：手动新增活动创建空白编辑态，不再默认填“新的活动 / 杭州 / 14:00-15:00”；地图点位只接收真实地点或用户填写地点；Agent fallback 没有真实 POI 时不再把目的地写成具体地点；批量补全和单段路线跳过没有明确起终点的活动；路线标题、地图卡、可访问标签和 fallback 路线步骤统一使用“活动标题 -> 地点名 -> 第 N 站 / 未命名活动”的显示名策略；客户端会把本机 SQLite 中已残留的旧默认占位活动规范化为空白活动，避免旧 demo 数据刷新后继续污染地图。
- 工作台第六批同类问题分析与修复：继续把“输入框堆叠、无意义状态、按钮语义混乱、红色选中态”归类为编辑焦点和任务语义问题。对应调整为：活动时间线卡片只保留摘要、排序和选择，具体字段集中到单一 `编辑活动` 面板；地图空状态删除与状态 chip 重复的“Day N 还没有地点”，并提供“添加空活动”入口；行程信息和路线手动覆盖从“保存信息/保存调整”改为“应用信息/应用调整”，避免与顶部同步状态冲突；活动卡、地图地点卡、路线卡和路线详情的选中态改用蓝色 `ring` 定位色，高德 polyline 选中颜色改为 `#435ee5`。
- 工作台第七批同类问题分析与修复：把“空白活动像真实内容”“导入/浏览等内部术语”“广场推荐页混入标签编辑”继续归类为任务语义和主次层级问题。对应调整为：空白手动安排统一显示为“待补全安排”，地图在已有空白安排但无地点时显示“Day N · X 项待补地点”和“安排还缺地点”；时间线空白卡只提示补地点后进入地图和路线，不再重复“未设置地点/时间/预算”；助手风格栏改为“当前风格 / 未选择风格 / 使用风格”；Skill 广场默认页改为发现与应用语义，推荐标识为“当前推荐”，卡片主动作改为“使用风格 / 移出行程”，标签编辑仅保留在草稿管理场景。
- 工作台第八批同类问题分析与修复：继续把“无意义状态”和“按钮语义不一致”归类为状态/命令混杂问题。对应调整为：顶部风格状态改为“当前风格 N 个”；批量路线按钮改为“补全 N 段路线”；Skill 广场推荐标识改为“推荐风格”；收藏数从按钮中拆出为“18 人收藏”，收藏按钮只保留“收藏 / 取消收藏”；部分填写活动摘要只展示已有信息，缺地点和时间时合并为“待补地点与时间”；导出 Markdown 不再把空白草稿误算为待计算交通，空白草稿标题导出为“待补全安排”。

### 下一批必须补齐

- 地图与行程还需要支持更强的定位能力：长行程下的活动筛选、点位搜索结果到地图的定位、路线段选中和冲突提示。
- 交通规划仍需继续产品化：需要补充路线段冲突提示、失败原因更细的用户修复建议和跨天长路线的总览筛选。
- Skill 广场需要完善从收藏、标签编辑、导入反馈到 Agent 实际读取的闭环。
- Skill 创作流程需要从“当前行程/外部游记/对话”生成 Skill 草稿，并按 Skill.md 标准格式校验、编辑、保存、发布。
- Agent 侧需要继续完善主 Agent 调用子 Agent、流式输出、终止、上下文隔离、会话历史读取、偏好总结和可理解的用户反馈。
- 最终交付前必须引入 product-design 的严格 audit/design-qa 流程，对产品完成度不足项返工。
